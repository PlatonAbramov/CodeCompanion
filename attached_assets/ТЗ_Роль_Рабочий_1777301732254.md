# Техническое задание: новая роль «Рабочий»

## 1. Цель

Добавить в систему ролей приложения новую роль **«Рабочий»** — для рядовых сотрудников на объектах, которые документируют ход работ через фотографии в листах выполнения, не имея доступа к коммерческой и финансовой информации.

Роль располагается ниже «Мастера» в иерархии и предназначена для людей, которые физически работают на разных объектах и должны иметь возможность подгружать фото-фиксации работ туда, где они в данный момент находятся.

## 2. Общие принципы

- Все изменения **строго аддитивные**: добавляется новая роль, новые проверки прав, при необходимости — новые поля. Существующая логика, экраны и API **не модифицируются**.
- Все доработки UI выполняются **в едином стиле с существующим приложением**: те же компоненты, цвета, иконки, навигация.
- Миграции БД — обратно совместимые, без простоя.
- Перед релизом — регрессионное тестирование существующих сценариев работы под другими ролями (Admin, Director, Master, Client).

## 3. Описание роли «Рабочий»

### Положение в иерархии
Иерархия становится: Администратор → Директор → Мастер → **Рабочий** → Клиент.

«Рабочий» — это полноценная роль учётной записи (не флаг), назначается админом или директором при создании / редактировании сотрудника на странице «Персонал».

### Что может «Рабочий»

#### Доступ к проектам
- **Видит список всех проектов** компании (в отличие от мастера, который видит только назначенные).
- В списке и карточке проекта отображаются **только название проекта и статус**. Все коммерческие данные (бюджет, согласованная сумма, клиент, контакты) — **скрыты**.
- Может зайти в любой проект.

#### Листы выполнения работ (Implementation Sheets)
- Может **просматривать листы выполнения** по любому проекту (структуру, описание работ, фото).
- Может **добавлять фотографии** в существующие листы выполнения.
- Может **удалять только те фотографии, которые загрузил он сам**. Удаление фотографий, загруженных другими пользователями, — запрещено.

#### Фотоконтроль автомобилей
- Если на сотрудника-рабочего дополнительно установлен флаг **«Водитель»** — он получает стандартные права водителя (проходить фотоконтроль закреплённого автомобиля, видеть свою историю и пробег). Флаг работает идентично тому, как он работает для мастера.

### Что «Рабочий» НЕ может

- Не видит и не имеет доступа: к расходам и чекам, доходам, авансам, вложениям собственника, бюджету, себестоимости, прибыли, любой финансовой отчётности.
- Не видит документы по проектам (договоры, КП, инвойсы, акты).
- Не видит и не редактирует справочники (клиенты, подрядчики, сотрудники, инструменты, автомобили).
- Не видит контактные данные клиентов и подрядчиков.
- Не создаёт и не редактирует сами листы выполнения (только добавляет/удаляет фото внутри уже существующих).
- Не меняет статус проекта.
- Не имеет прав в управлении пользователями, аудит-журнале, сессиях.

## 4. Требования к функционалу фотографий в листах выполнения

Чтобы корректно реализовать правило «удалять только свои фото», нужно:

- При загрузке каждой фотографии система должна сохранять метаданные: **кто загрузил** (user_id), **когда загрузил** (timestamp). Если эти поля уже хранятся — использовать существующую логику.
- В UI листа выполнения у каждой фотографии для пользователя-рабочего:
  - кнопка «удалить» **активна и видима только** на тех фото, которые загрузил он сам;
  - на чужих фото кнопка удаления отсутствует или неактивна.
- На сервере при запросе на удаление фото должна выполняться **двойная проверка**: (1) у пользователя роль «Рабочий», (2) `photo.uploaded_by == request.user.id`. Если не совпадает — вернуть 403 / запретить операцию.
- Для других ролей (Admin, Director, Master в своих проектах) поведение остаётся прежним — они могут удалять любые фото.

## 5. UI и навигация в приложении

При входе под ролью «Рабочий»:

- В нижнем меню / основной навигации **скрываются**: разделы и пункты, к которым у роли нет доступа (расходы, доходы, отчёты, документы, справочники и т. д.).
- Видимыми остаются:
  - Раздел «Проекты» — упрощённый вид (только список с названиями и статусами).
  - Раздел «Автомобили» — **только если** на сотруднике установлен флаг «Водитель», иначе скрыт.
  - Минимальный профиль / настройки аккаунта.
- В карточке проекта для рабочего **доступна только вкладка «Листы выполнения»** (или эквивалент в текущей структуре). Остальные вкладки (расходы, документы, финансы) — скрыты.
- При добавлении фото в лист — **обязательно отображать контекст**: к какому проекту и какому листу выполнения добавляется фото (хлебные крошки или явный заголовок). Это нужно потому, что рабочий видит все проекты сразу и может ошибиться при выборе.

## 6. Технические требования (безопасность)

- **Все проверки прав выполняются и на клиенте, и на сервере.** Скрытие элементов в UI — это удобство, а не защита. API-эндпоинты должны самостоятельно проверять роль пользователя и отказывать в доступе к закрытым данным даже при прямых запросах в обход интерфейса.
- API проектов для пользователя-рабочего должен возвращать **усечённый объект**: только разрешённые поля (название, статус). Финансовые поля либо исключаются из ответа, либо явно null/обнуляются.
- API расходов, доходов, документов, отчётов — для рабочего возвращают 403.
- Все ключевые действия рабочего логируются с timestamp и user_id: загрузка фото, удаление фото, вход в систему.

## 7. Изменения на странице «Персонал»

- В выпадающем списке ролей при создании / редактировании сотрудника появляется новый вариант **«Рабочий»**.
- В списке сотрудников у записей с ролью «Рабочий» отображается соответствующая метка/бейдж в общей стилистике.
- Назначить и снять роль «Рабочий» может **только админ или директор**.
- Если у сотрудника-рабочего стоит флаг «Водитель» — флаг сохраняется при назначении / снятии роли (поведение флага не зависит от роли).

## 8. Совместимость с существующим приложением

- Существующие роли и их права **не затрагиваются**.
- Новая роль реализуется как ещё одна позиция в существующей системе ролей, без переписывания базовой архитектуры прав.
- Все экраны, которые рабочий не должен видеть, скрываются за **проверками прав**, а не за переписыванием самих экранов.

## 9. Что НЕ входит в задачу

- Привязка рабочего к конкретным проектам (он видит все по дизайну).
- Уведомления о добавлении / удалении фото — не требуются.
- Лимиты на количество загружаемых фото — не требуются.
- Согласование загруженных фото мастером / директором перед публикацией — не требуется.
- Возможность рабочего комментировать листы выполнения текстом — не требуется (только фото).
