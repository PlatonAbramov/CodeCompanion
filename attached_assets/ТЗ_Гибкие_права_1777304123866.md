# Техническое задание: гибкая настройка прав ролей и пользователей

## 1. Цель

Добавить в панель администратора функцию **гибкого управления правами**:

- Для каждой роли (Админ, Директор, Мастер, Рабочий, Клиент) администратор может включать и выключать конкретные права через галочки.
- Для любого отдельного сотрудника администратор может выдать **персональные права-исключения** — включить дополнительно или выключить из тех, что положены по роли.
- При первом запуске у каждой роли выставлен набор прав по умолчанию, который **в точности соответствует тому, что роль может сегодня**. Дальше администратор редактирует на своё усмотрение.

Цель — дать гибкость без необходимости создавать новые роли под каждую вариацию.

## 2. Общие принципы

- Все изменения **строго аддитивные**: добавляется новая страница в админ-панели и логика управления правами. Существующие экраны и поведение системы **не модифицируются**.
- На момент релиза поведение системы **не должно измениться** ни для одного пользователя. Изменения начинаются только после ручной правки админом.
- Перед релизом — регрессионное тестирование под всеми ролями: убедиться, что без правок настроек всё работает как раньше.
- Все доработки UI — в едином стиле с существующим приложением.

## 3. Логика прав

### Двухуровневая модель

Итоговое право пользователя определяется по правилу:

- если для пользователя задано **персональное переопределение** — действует оно;
- иначе действует значение, заданное **на уровне его роли**.

Это значит:
- **Уровень роли** задаёт значение «по умолчанию для всех пользователей этой роли».
- **Уровень сотрудника** может точечно для конкретного человека либо включить недостающее право, либо выключить ненужное.

### Состав прав

Все возможные права в системе сгруппированы по категориям. Минимум — те же категории, что в существующей матрице прав:

- Пользователи и доступ
- Проекты
- Финансы
- Документы и листы выполнения
- Справочники
- Автомобили и фотоконтроль
- Система

Полный список конкретных прав, которые попадут в настройки, — по существующей матрице прав ролей. Все права, которые сейчас фактически проверяются в системе, должны быть доступны для настройки.

## 4. UI в панели администратора

### Новый раздел «Права и доступ»

Новая страница в существующей админ-панели. Доступ — **только у роли «Админ»**.

Страница состоит из двух разделов: **«По ролям»** и **«По сотрудникам»**.

### Раздел «По ролям»

- Выбор роли: Админ / Директор / Мастер / Рабочий / Клиент.
- После выбора отображается список всех прав, сгруппированных по категориям.
- Каждое право — отдельная строка с:
  - понятным названием на русском (например, «Создавать расходы», «Видеть все проекты»),
  - кратким пояснением в одну строку под названием,
  - переключателем on/off.
- Возможность массовых действий по категории (включить всё / выключить всё) — желательно, на усмотрение разработчиков.
- Кнопка **«Сбросить к значениям по умолчанию»** — возвращает все права роли к стандартному набору, заложенному при первом запуске функции.
- Изменения сохраняются явно (кнопка «Сохранить»), с предварительным подтверждением: «Вы изменили N прав для роли X. Сохранить?»

### Раздел «По сотрудникам»

- Поиск и выбор конкретного сотрудника.
- После выбора отображается список всех прав, сгруппированных по категориям.
- У каждого права — **три состояния**:
  - **По умолчанию** — наследуется от роли (показывается текущее эффективное значение из настроек роли).
  - **Включено** — персонально включено для этого сотрудника.
  - **Выключено** — персонально выключено для этого сотрудника (явный запрет).
- Рядом с каждым правом — пояснение, что и откуда сейчас действует:
  - «наследуется от роли Мастер: включено»
  - «персонально включено»
  - «персонально выключено»
- Кнопка **«Сбросить персональные настройки»** — снимает все индивидуальные переопределения, сотрудник снова работает строго по правам своей роли.
- Изменения сохраняются явно, с подтверждением.

### Журнал изменений

Внутри раздела «Права и доступ» — встроенный журнал последних изменений: кто (админ), когда, для какой роли или какого сотрудника, какое право включил или выключил, прежнее и новое значение. Минимум 100 последних записей. Полный аудит — в общем аудит-журнале системы.

## 5. Стандартные значения по умолчанию

При первом запуске функции для каждой роли выставляется набор прав, **строго соответствующий текущей матрице прав** (Админ, Директор, Мастер, Рабочий, Клиент + флаг «Водитель»).

Контрольная проверка перед релизом: войти под каждой ролью на тестовом стенде и убедиться, что все экраны и действия работают так же, как до внедрения функции — без необходимости что-то менять в новых настройках.

## 6. Применение прав в системе

- Все проверки прав выполняются **и на сервере, и в UI**. UI скрывает / отключает недоступные элементы для удобства, сервер — основная защита от обхода через прямые запросы.
- Изменения, сделанные администратором на странице «Права и доступ», должны применяться у пользователя в течение разумного времени (например, при следующем запросе или после повторного входа).
- При попытке выполнить действие без права — сервер возвращает понятную ошибку доступа.

## 7. Защита от опасных конфигураций

Чтобы администратор случайно не заблокировал систему, нужны защитные механизмы:

- **Последний администратор** — нельзя выключить право управления правами у последнего активного пользователя с ролью Админ. UI должен это пресекать с понятным сообщением.
- **Самоблокировка** — если администратор пытается выключить себе ключевые админские права (например, доступ к разделу «Права и доступ»), UI требует явного подтверждения и предупреждает о последствиях.
- **Системные права** — некоторые базовые права (например, вход в систему, смена своего пароля) не должны отображаться в UI на редактирование вовсе. Их состав определяют разработчики и согласовывают с заказчиком.

## 8. Поведение при добавлении новых прав в будущем

Когда в системе появится новая функциональность с новым правом:

- Новое право автоматически появляется в разделе «Права и доступ» в нужной категории.
- Для каждой роли при добавлении нового права заранее определяется разумное значение по умолчанию (например: для Админа — включено, для остальных — на согласование при разработке этой новой функции).
- Существующие персональные переопределения сотрудников **не затрагиваются**.

## 9. Совместимость и безопасность

- Существующие роли и привязки сотрудников к ролям **сохраняются как есть**.
- Существующие флаги (например, «Водитель») продолжают работать как раньше — это отдельный механизм поверх системы прав, его не трогаем.
- Все действия по изменению прав логируются: кто, когда, что изменил, прежнее и новое значение. Логи доступны через журнал изменений и общий аудит.

## 10. Что НЕ входит в задачу

- Создание собственных кастомных ролей с нуля (только редактирование существующих) — не требуется.
- Группы пользователей с общими переопределениями — не требуется, переопределения только индивидуальные.
- Временные права (с автоматическим истечением) — не требуются.
- Права на конкретные объекты (например, «этот сотрудник видит только эти 3 проекта») — не требуются, права действуют глобально в рамках системы.
- Импорт / экспорт настроек прав — не требуется.
