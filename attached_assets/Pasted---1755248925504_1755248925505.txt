ТЗ: Полностью автоматические миграции БД при старте сервиса + автосид директора
Цели

Убрать падения Redeploy из-за миграций.

Перенести миграции с «платформенного шага Replit» на старт сервиса.

Делать предобработку данных (backfill/чистка) и схему (идемпотентно) автоматически.

Автоматически создавать/обновлять учётку директора (platonabramov90@gmail.com, login: platonabramov90, пароль 123456, роль director) — с хэшированным паролем.

Не менять бизнес-логику приложения.

Итог, который должен быть у команды

Один стартовый скрипт (entrypoint) — он запускается при каждом деплое/рестарте и сам:

ждёт, пока БД поднимется (retries/backoff);

берёт advisory-lock в Postgres, чтобы миграцию делал только один экземпляр;

последовательно выполняет 4 шага: preflight → backfill → миграция схемы → сид директора;

пишет читабельные логи (start/ok/fail с кодами и временем);

при успехе снимает lock и стартует приложение;

по флагу AUTO_MIGRATE=0 пропускает этап миграций (аварийный обход).

Миграционные SQL/скрипты — идемпотентные (повторный запуск ничего не ломает).

Replit Deploy не запускает свои «database setup/migrations».

Архитектура решения
1) Отключить платформенные миграции

В Replit/Deploy убрать любые автоматические шаги «migrate» (build/run hooks ORM и т.п.).

Оставить только запуск нашего entrypoint.

2) Новый стартовый сценарий (entrypoint)

Задачи entrypoint’а:

Прочитать DATABASE_URL из Secrets.

Сделать wait-for-db с экспоненциальной задержкой (например, до 60–90 сек суммарно).

Выполнить блок DB BOOTSTRAP под одним advisory-lock (например, ключ 748392), чтобы параллельные инстансы не лезли одновременно.

По шагам:

Preflight (только чтение): вывести в лог, есть ли NULL в обязательных колонках users и «сироты» в sessions/login_attempts/mfa_totp.

Backfill/clean: задать дефолты в users, заполнить текущие NULL (даты/флаги/логин), удалить «сироты», временно блокировать пользователей без валидного хэша пароля.

Migration: применить изменения схемы (ослабления/добавления → ужесточения/FK), всюду использовать проверки существования (IF EXISTS/NOT EXISTS/проверки в DO-блоках).

Seed директора: создать/обновить учётку директора (upsert) с хэшом пароля (bcrypt/argon2). Если role — enum, автоматически добавить director, если его нет.

Вести чёткие логи: старт каждого шага, итог, длительность. В случае ошибки — печатать текст и выходить с кодом 1 (пусть оркестратор перезапустит).

Если AUTO_MIGRATE=0 — пропускаем весь блок DB BOOTSTRAP, сразу стартуем приложение.

3) Идемпотентность и устойчивость

Все операции на схеме/данных писать с проверками существования (колонок/ограничений/значений enum).

Опасные операции заворачивать в транзакции; порядок: ослабить/добавить → backfill → ужесточить → навесить FK.

На каждое добавление FK — предварительная чистка сирот.

Для несовпадающих имён констрейнтов использовать проверку по pg_constraint и DO-блоки.

4) Seed директора (автоматически)

Выполняется каждый старт (идемпотентно):

если пользователя с email platonabramov90@gmail.com нет — создать с login: platonabramov90, роль director, is_blocked = false, пароль как хэш;

если есть — убедиться, что login='platonabramov90', role='director', is_blocked=false; при расхождении — привести в нужный вид.

Если расширение pgcrypto недоступно — хэшируйте на стороне приложения (bcrypt/argon2) до отправки в БД.

5) Переменные окружения

DATABASE_URL — строка подключения к Postgres.

AUTO_MIGRATE=1 — по умолчанию включено; 0 — аварийно пропустить.

DB_LOCK_KEY=748392 — ключ advisory-lock (можно поменять).

DB_BOOT_RETRIES=15, DB_BOOT_BACKOFF_MS=2000 — политика ожидания БД.

(если нужно) DIRECTOR_EMAIL, DIRECTOR_LOGIN, DIRECTOR_PASSWORD — чтобы не хардкодить.

6) Логи/мониторинг

Метки: [DBBOOT] WAIT/LOCK/PREFLIGHT/BACKFILL/MIGRATE/SEED/UNLOCK/DONE + время выполнения.

Ошибки краткие и понятные (что делали, на чём упали).

Добавить счётчики: кол-во очищенных «сирот», кол-во исправленных NULL, флаг «директор создан/обновлён».

Что надо сделать разработчикам (списком задач)

Создать модуль «db-bootstrapper» в репозитории (папка db/auto/ или scripts/):

реализовать команды: wait-for-db, preflight, backfill, migrate, seed_director;

каждая команда — идемпотентна; объединить их в общий раннер «run-on-boot».

Встроить advisory-lock вокруг всего блока миграций (до запуска приложения).

На Postgres: pg_advisory_lock(KEY) / pg_advisory_unlock(KEY).

Перенастроить команду запуска сервиса так, чтобы она вызывала наш entrypoint:

entrypoint: DB BOOTSTRAP (если AUTO_MIGRATE=1) → старт приложения.

Никаких ORM-миграций в build/run шагах Replit.

Реализовать сид директора как часть «run-on-boot».

Пароль — только хэш (bcrypt cost ≥ 12 или argon2id).

Обработать случай enum-роли (автодобавление director).

Добавить конфигурационные флаги (см. раздел «Переменные окружения»).

Значения — через Replit Secrets.

По умолчанию включён AUTO_MIGRATE.

Журналирование: печатать этапы и итоги; в случае ошибки — понятный текст и завершение с ненулевым кодом.

(Опционально) ORM-миграции: если в проекте используется ORM, запускать её «deploy-миграции» после наших SQL-шагов, только если они тоже идемпотентны и не конфликтуют.

Критерии приёмки

На «чистом» окружении сервис с AUTO_MIGRATE=1 поднимается без ручных действий: ждёт БД, мигрирует, сидит директора, стартует.

На «грязной» БД (с NULL и «сиротами») первый старт сам:

заполняет NULL, удаляет «сирот»,

делает изменения схемы (ослабления/добавления → ужесточения/FK),

создаёт/обновляет директора,

и запускает приложение без падения.

Повторный старт (несколько раз подряд) не ломает схему, ничего не дублирует (идемпотентность).

При параллельном старте двух инстансов миграции выполняются строго один раз (advisory-lock работает).

В логах видны этапы и длительность; при ошибке — понятное сообщение.

Replit Redeploy больше не падает на «Database setup/migrations».
