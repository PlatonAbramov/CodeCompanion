1) История изменений («кто что поменял»)
Цель

Фиксировать все значимые изменения в системе с указанием: кто, что, где и когда изменил; уметь просматривать историю в контексте проекта, позиции, платежа, документа, инструмента.

Что пишем в историю

Объект: тип (проект, позиция листа реализации, расход, аванс, документ, инструмент, подрядчик, заказчик, пользователь).

Идентификатор объекта.

Действие: создание / изменение / удаление / загрузка файла / передача инструмента / возврат / смена статуса / изменение процента выполнения / отметка «выполнено».

Поля «было → стало» (для изменения).

Кто сделал (имя пользователя + его роль).

Когда сделал (дата и время).

Привязка к проекту (если применимо).

Где показываем

В карточке проекта — вкладка «История»: все события по этому проекту и связанным объектам, с фильтрами по типу, пользователю, дате.

В карточке объекта (позиция, платеж, документ, инструмент) — мини-лента истории по конкретному объекту.

Права

Администратор: видит всю историю.

Прораб: историю по своим проектам.

Мастер: историю по своим проектам (только чтение).

Заказчик: не видит (или видит только отмеченные «для заказчика» события — по решению).

Приёмка:

Любое изменение создаёт запись в истории.

История корректно фильтруется и сортируется по времени.

Для изменения прогресса/статуса фиксируются «было → стало».

Удаление объекта отображается с указанием удалившего.

2) Архив проектов
Цель

Скрывать завершённые/замороженные проекты из списка активных, не теряя данные и доступ к ним.

Правила
Статусы проекта: «в работе», «завершён», «архив».
Перевод в архив — вручную из карточки проекта любой ролью с правом (только Администратор).
Проекты в архиве скрываются из «активных» списков и дашборда по умолчанию.
Отдельный список «Архив» с поиском и фильтрами.
Проект из архива можно вернуть в «в работе».

Поведение
Перевод в архив не меняет финансовые данные, документы и историю.
Нельзя редактировать архивный проект, кроме действия «вернуть в работу» и просмотра (по решению — Администратору можно редактировать).

Приёмка
Проект успешно переводится в архив и пропадает из активных.
В архивном списке проект виден, открывается в режиме просмотра.
Возврат из архива возвращает проект в обычные списки.

3) «Личный кабинет» заказчика (веб/мобайл-просмотр)
Цель
Дать заказчику удобный просмотр своего проекта: этапы/прогресс, фото/видео, документы «для заказчика», базовая информация.

Доступ
Заказчик входит под своим логином (Который создает для него Администратор.
Видит только свои проекты.
Нет возможности редактировать, загружать и удалять.

Что видит заказчик в проекте
Название проекта, адрес/описание (если есть), даты, общий прогресс, статус.

Лист реализации в режиме просмотра:
Позиции с прогрессом в процентах и ценами.
Отметка «выполнено» (только просмотр).
Фото/видео-миниатюры по позициям.
По клику — модалка с увеличением и пролистыванием.
Документы, помеченные как «для заказчика», со скачиванием/просмотром.
История публичных событий (если включено): обновление этапов, загрузка «публичных» медиа.

Видимость
У фото и документов есть флаг «показывать заказчику».
Позиции листа реализации можно помечать как «видно заказчику» или скрывать.

Приёмка

Заказчик видит только свои проекты и только помеченные материалы.
Мобильная верстка удобна: список этапов, лист реализации, просмотр медиа внутри страницы.
Попытка доступа к чужому проекту — отказ, они ему просто не показываются.

5) Аналитика и отчёты
Дашборд по проектам (Все должно быть внутри одной кнопки чтобы не перегружать поля

Цель: быстрый обзор бизнеса.
Карточки/виджеты:
Количество активных проектов / архивных.
Общая контрактная стоимость активных.
Сумма полученных платежей (по заказчикам).
Сумма расходов (факт).
Фактическая прибыль = платежи минус расходы (по активным).
Средний прогресс по активным проектам.

Списки:

Топ проектов по бюджету, по прибыли, по просрочке по срокам.
Фильтры: период (месяц/квартал/год), статус проекта.

Приёмка:

Все суммы сходятся с данными проектов.

Прогресс считается корректно (среднее или заданная методика).

Фильтры влияют на показатели.

Отчёт по подрядчикам

Цель: видеть обязательства и оплату подрядчикам.

По каждому подрядчику:

Количество проектов.

Сумма по договорам с подрядчиком (или оценка работ).

Выплачено.

Осталось к выплате.

Доля выполненных работ (по связным позициям/этапам проекта).

Итоги по всем подрядчикам.

Фильтры: подрядчик, период, проект, статус.

Приёмка:

Сводные цифры равны сумме развернутых данных.

«Осталось к выплате» = договорная сумма минус выплачено.

Доля выполненных рассчитывается от закреплённых за подрядчиком работ (методика фиксируется один раз и не меняется без согласования).

Отчёт по заказчикам

Цель: управляемость платежной дисциплиной.

По каждому заказчику:

Количество проектов.

Сумма по контрактам.

Оплачено.

Остаток к оплате.

Статус платежной дисциплины:

«В срок» — платежи по графику.

«Задержки» — есть просроченные платежи.

Выводить список просроченных платежей: проект, сумма, дата, на сколько дней просрочка.

Фильтры: заказчик, период, статус.

Приёмка:

Итоги совпадают с платежами по проектам.

Просрочки правильно считаются относительно графика.

Заказчики сортируются по сумме просрочки/остатка.

Отчёт по инструментам

Цель: контроль наличия и потерь.

Показатели:

Сколько «на руках», у кого именно.

Сколько «в наличии».

Списанные.

Движение за период: передано, возвращено.

Предполагаемые потери: долго «на руках» без возврата (порог задаётся).

Перечни:

Инструменты без движения слишком долго.

Инструменты с частыми поломками/списаниями.

Фильтры: период, статус, ответственный.

Приёмка:

Количество «на руках» совпадает со сводом по людям.

Списанные не попадают в «в наличии».

Движение отражается корректно.

Общие требования к аналитике

Все цифры должны совпадать с первичными данными (проекты, платежи, расходы, лист реализации, инструменты).

В каждой метрике понятная формула расчёта и единицы (валюта — AED).

Экспорт ключевых отчётов в PDF (как минимум дашборд-сводка и три отчёта).

Даты и суммы отображаются в едином формате.
