## Цель
Привести визуальный облик приложения к уровню современного мобильного продукта (Revolut / Notion / Linear): согласованная типографика, правильная цветовая система, профессиональная компоновка финансовых данных, нативная мобильная навигация.

---

## Общие правила для всех задач (обязательно к исполнению)

1. **Отдельная ветка и отдельный PR на каждую задачу.** Именование: `feature/design-<номер>-<короткое-название>`. Смешивать задачи в одном PR запрещено.
2. **Скриншоты ДО и ПОСЛЕ** в описании PR для каждого затронутого экрана.
3. **Не менять бизнес-логику.** Запрещено: править API, структуру БД, формулы расчёта, права доступа. Только визуальная и структурная вёрстка.
4. **Не ломать существующие сценарии.** После каждой задачи прогнать чек-лист тестовых сценариев (см. ниже). Все кнопки, ссылки, переходы должны работать как раньше.
5. **Правило отката.** Если задача:
   - ломает хотя бы один тестовый сценарий,
   - вызывает регресс на продакшне,
   - не выглядит лучше, чем было (subjective check при код-ревью),

   разработчик делает `git revert` всей ветки и пробует альтернативу из раздела «Альтернативы». Запрещено оставлять полурабочее решение.
6. **Адаптивность.** Всё тестировать на ширинах: 360px (маленький мобильный), 414px (iPhone), 768px (планшет). Нигде не должно быть переполнения или обрезанных элементов.
7. **Без правок без ТЗ.** Если разработчик видит ещё что-то, что хочется улучшить — записать в отдельный issue, **не делать в рамках текущей задачи**. Иначе невозможно отследить, что именно сломалось.
8. **Код-ревью обязателен** перед мержем.

# Задача 1. Глобальная типографика и tabular-nums

## Проблема
Нет фиксированной шкалы размеров шрифта — в разных местах используются произвольные значения. Цифры в финансовой сводке («189 200», «52 550», «366 883») разной ширины — колонки плывут. Заголовки используют `font-bold`, что на мобильном выглядит «жирно-примитивно».

## Цель
Зафиксировать типографическую шкалу, включить табличные цифры везде, где выводятся суммы, нормализовать начертания.

## Файлы
- `tailwind.config.ts`
- `client/src/index.css`
- Компоненты с числовыми значениями (финансовая сводка, карточки статистики, списки расходов).

## Реализация

1. Подключить шрифт Inter через `index.css`:
   ```css
   @import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&display=swap');
   ```
2. В `tailwind.config.ts` задать `fontFamily.sans` с Inter в начале.
3. Зафиксировать шкалу размеров (использовать только её, никаких `text-[17px]`):
   - `text-xs` (12px) — служебные подписи, даты, мелкие метки.
   - `text-sm` (14px) — вторичный текст.
   - `text-base` (16px) — основной текст, названия в списках.
   - `text-lg` (18px) — заголовки секций.
   - `text-2xl` (24px) — заголовки страниц.
   - `text-3xl` (30px) — главные цифры (прибыль, бюджет).
4. Начертания: заголовки — `font-semibold`, не `font-bold`. Жирнее — только для крупных чисел.
5. Глобально в `index.css`:
   ```css
   :root { font-feature-settings: "cv11", "ss01"; -webkit-font-smoothing: antialiased; }
   ```
6. Все числовые значения (суммы, количества, проценты) обернуть классом `tabular-nums`. Проверить по файлам:
   - `client/src/pages/ProjectDetails.tsx` (финансовая сводка)
   - `client/src/pages/Dashboard.tsx` (карточки статистики)
   - `client/src/components/ProjectCard.tsx`
   - `client/src/pages/Expenses.tsx` и аналогичные.
7. Для крупных сумм — единица валюты (`AED`) отделить визуально: либо оттенком (`text-muted-foreground`), либо на 2 размера меньше в том же контейнере. Пример:
   ```tsx
   <span className="text-3xl font-semibold tabular-nums">
     81 846 <span className="text-xl text-muted-foreground font-normal">AED</span>
   </span>
   ```

## Критерии приёмки
- В финансовой сводке цифры в колонке выровнены по разрядам идеально (разряды тысяч под разрядами тысяч).
- Никаких `text-[NNpx]` в коде — всё через Tailwind-шкалу.
- Все заголовки — `font-semibold`, не `font-bold`.
- Визуально текст на всех экранах выглядит согласованно.

## Откат и альтернативы
- Ветка `feature/design-1-typography`.
- Если Inter не загружается в Capacitor-сборке — встроить шрифт локально через `@fontsource/inter`.
- Если задача затрагивает слишком много файлов и что-то ломается — откатить и делать по одному разделу за раз.

# Задача 2. Цветовая система и CSS-переменные

## Проблема
На одном экране используются: синий brand, зелёный для прибыли и статусов, красный для расходов и авансов, голубой для акцентов, серый. Нет правила, когда какой цвет применяется. Красный применяется к обычным суммам расходов, что психологически неверно (расход — это не ошибка).

## Цель
Зафиксировать палитру и правила её использования. Все цвета — только через CSS-переменные.

## Файлы
- `client/src/index.css` (секция `:root` и `.dark`).
- `tailwind.config.ts` (mapping переменных на классы).
- Все места, где используется hex-цвет или Tailwind-цвет вроде `text-red-600`, `text-green-500` напрямую.

## Реализация

1. В `index.css` определить палитру (если ещё не определена по стандарту shadcn):
   ```css
   :root {
     --background: 0 0% 100%;
     --foreground: 222 47% 11%;
     --muted: 210 40% 96%;
     --muted-foreground: 215 16% 47%;
     --card: 0 0% 100%;
     --border: 214 32% 91%;
     --primary: 217 91% 60%;          /* brand-синий */
     --primary-foreground: 0 0% 100%;
     --success: 142 71% 45%;          /* зелёный — только для позитивных дельт */
     --success-foreground: 142 76% 96%;
     --danger: 0 72% 51%;             /* красный — ТОЛЬКО ошибки и убытки */
     --danger-foreground: 0 0% 100%;
     --warning: 38 92% 50%;           /* жёлтый — предупреждения, просрочки */
     --neutral-amount: 222 47% 11%;   /* обычные суммы — тёмно-серый */
   }
   ```
2. Задокументировать правила в `client/src/index.css` комментарием или в отдельном `DESIGN_TOKENS.md`:
   - **primary (синий)** — интерактивные элементы: кнопки, ссылки, активная навигация, фокус.
   - **success (зелёный)** — **только** позитивные дельты, статус «Завершён», галочки успеха. **Не** для обычных сумм.
   - **danger (красный)** — **только** убыток, ошибки, удаление, просрочка. **Не** для расходов и авансов.
   - **warning (жёлтый)** — приближение к лимиту, просрочка аванса.
   - **neutral (тёмно-серый)** — все нейтральные суммы: стоимость договора, авансы, расходы.
3. Пройти по коду, заменить прямые Tailwind-цвета (`text-red-500`, `bg-green-100`) на семантические токены.
4. В `tailwind.config.ts` убедиться, что все цвета прокинуты как `bg-primary`, `text-success`, `text-danger` и т. д.

## Критерии приёмки
- В репозитории нет hex-цветов в JSX/TSX-файлах (проверить grep: `#[0-9a-fA-F]{3,6}`).
- В репозитории нет прямых Tailwind-цветов вида `text-red-600`, `bg-green-500` в компонентах страниц (только в базовых UI-компонентах shadcn). Grep: `(text|bg|border)-(red|green|blue|yellow|orange)-[0-9]`.
- Все цвета идут через CSS-переменные и семантические классы.

## Откат и альтернативы
- Ветка `feature/design-2-color-system`.
- Если какой-то сторонний компонент жёстко завязан на цвет — оставить точечное исключение, задокументировать.
- Если миграция затрагивает слишком много — делать по страницам в отдельных PR, но всё равно в рамках этой ветки-задачи (suball-PR на большую ветку).

# Задача 3. Переработка финансовой сводки проекта

## Проблема
На странице проекта финансовая сводка:
- «Взятые авансы 52 550 AED» — красным. Это нейтральный факт, не убыток.
- «Расходы на проект 366 883 AED» — красным. Расход — норма, не ошибка.
- Главная цифра проекта (прибыль/убыток) **не показана** или не выделена.
- Кнопки `+` и `✎` в каждой строке создают визуальный шум.
- Нет чёткой иерархии — все строки выглядят одинаково.

## Цель
Превратить сводку в читаемую финансовую панель, где первым делом видна главная цифра (прибыль/убыток), затем структурированно — все составляющие.

## Файлы
- `client/src/pages/ProjectDetails.tsx` (или где рендерится финансовая сводка).
- `client/src/components/FinancialSummary.tsx` (если вынесен) или создать.

## Реализация

1. **Сверху сводки — главная цифра**: прибыль/убыток проекта крупно.
   ```tsx
   <div className="bg-card rounded-2xl p-5 border">
     <div className="text-xs text-muted-foreground uppercase tracking-wide">
       Прибыль проекта
     </div>
     <div className={cn(
       "text-4xl font-semibold tabular-nums mt-1",
       profit >= 0 ? "text-success" : "text-danger"
     )}>
       {profit >= 0 ? '+' : ''}{formatCurrency(profit)} <span className="text-2xl font-normal text-muted-foreground">AED</span>
     </div>
     <div className="text-sm text-muted-foreground mt-1">
       {profitPercent}% от стоимости договора
     </div>
   </div>
   ```
2. **Далее — детализация в виде списка-выписки**:
   - `divide-y divide-border` между строками вместо больших gap.
   - Каждая строка: слева label (`text-sm text-muted-foreground`), справа сумма (`text-base font-medium tabular-nums`).
   - Все нейтральные суммы — цвет `text-foreground`, **не красный, не зелёный**.
   - Красный — только итоговый убыток. Зелёный — только итоговая прибыль.
3. **Редактирование — по тапу на строку**, не через `+/✎` в строке. Тап → открывается bottom sheet с полем и кнопкой «Сохранить». Маленькая иконка `chevron-right` справа от суммы — индикатор интерактивности.
4. **Группировка**: «Доходы» (стоимость договора, аванс от заказчика) и «Расходы» (взятые авансы, вложения, расходы на проект) — под отдельными подзаголовками внутри одной карточки.
5. **Единицы валюты** — оформить как описано в Задаче 1: основная цифра + приглушённое `AED` рядом.

## Критерии приёмки
- Главная цифра (прибыль/убыток) видна сразу при открытии страницы, без скролла.
- Красный цвет встречается только в итоговой строке при убытке.
- Цифры в колонке выровнены идеально (`tabular-nums`).
- Редактирование работает через тап на строку (bottom sheet).
- Все существующие действия (добавить аванс, добавить расход) сохранены, просто перенесены внутрь соответствующих модалок.

## Откат и альтернативы
- Ветка `feature/design-3-financial-summary`.
- Если bottom sheet для редактирования сложно подключить — оставить временно модалку через `<Dialog>` shadcn, UX почти одинаковый.
- Если что-то сломает работу расчётов — откат, логика не трогается.

# Задача 4. Замена hamburger-меню на tab-bar

## Проблема
Внизу экрана hamburger-кнопка «Меню», открывающая bottom sheet со списком разделов. Это устаревший паттерн, пользователь каждый раз делает лишний тап.

## Цель
Реализовать нативную мобильную нижнюю навигацию с 4–5 основными разделами.

## Файлы
- Новый компонент: `client/src/components/layout/BottomNav.tsx`.
- Основной layout: `client/src/App.tsx` или `client/src/components/layout/MobileLayout.tsx`.
- Старый компонент меню — отрефакторить (не удалять, а превратить в «More» bottom sheet).

## Реализация

1. Основные 4 таба: **Главная** (Home), **Проекты** (FolderKanban), **Финансы** (Wallet), **Ещё** (MoreHorizontal).
2. Компонент таб-бара:
   ```tsx
   <nav className="fixed bottom-0 inset-x-0 bg-background border-t border-border
                   pb-[env(safe-area-inset-bottom)] z-40">
     <div className="grid grid-cols-4">
       {tabs.map(tab => (
         <button key={tab.id} onClick={...}
           className={cn(
             "flex flex-col items-center gap-1 py-2.5 transition-colors",
             active ? "text-primary" : "text-muted-foreground"
           )}>
           <tab.Icon className="w-5 h-5" strokeWidth={1.75} />
           <span className="text-[11px] font-medium">{tab.label}</span>
         </button>
       ))}
     </div>
   </nav>
   ```
3. **Safe area**: обязательно `pb-[env(safe-area-inset-bottom)]` — чтобы на iPhone не уезжало под home indicator.
4. **Активное состояние**: иконка + текст в `text-primary`, под иконкой можно добавить тонкий индикатор-полоску `h-0.5 w-6 rounded-full bg-primary` над табом (опционально).
5. **Таб «Ещё»** открывает тот же bottom sheet, что сейчас по hamburger — с пунктами «Подрядчики», «Заказчики», «Инструменты», «Персонал», «Сотрудники», «Настройки», «Выход». Компонент меню переиспользовать, не переписывать.
6. **Отступ контента**: в layout добавить `pb-20` на основной контент, чтобы таб-бар не перекрывал последние элементы страниц.
7. **Haptic feedback**: на тап по табу — `Haptics.impact({ style: ImpactStyle.Light })` через `@capacitor/haptics` (если не установлен — `npm install @capacitor/haptics`).

## Критерии приёмки
- Таб-бар виден на всех основных страницах.
- Активный таб визуально отличается.
- На iPhone нижний край корректно обходит home indicator.
- На Android нижняя навигационная панель системы не перекрывает таб-бар.
- Все разделы, которые были в старом меню, доступны через «Ещё».
- Никакой контент на страницах не прячется под таб-баром.

## Откат и альтернативы
- Ветка `feature/design-4-tab-bar`.
- Если активное состояние визуально спорит с чем-то — смягчить (убрать полоску-индикатор, оставить только цвет иконки).
- Если Haptics конфликтует с веб-сборкой — обернуть в try/catch и проверку Capacitor-рантайма.
- При проблемах — откат, остаётся старое меню.

# Задача 5. Бейджи вместо цветного текста для статусов

Унифицировать все статусы через компонент `<Badge>` с фоновой заливкой.

- `client/src/components/ui/badge.tsx` (shadcn, должен существовать).

1. Если компонента `Badge` нет — сгенерировать через `shadcn add badge`.

   ```tsx
   const statusStyles = {

     'Завершён':   'bg-emerald-50 text-emerald-700 ring-emerald-200',
     'Приостановлен': 'bg-amber-50 text-amber-700 ring-amber-200',
   ```

   ```tsx
   <span className={cn(
   ```

   - Карточки проектов на Dashboard.

- Нигде нет «голого» цветного текста в качестве статуса.

- Переключение языка RU/EN не ломает вёрстку бейджей (длинные английские названия помещаются).

## Откат и альтернативы
- Ветка `feature/design-5-status-badges`.
- Если где-то бейдж ломает вёрстку (длинный текст переносится) — уменьшить `text-[10px]` или обрезать текст.
- Минимальный риск, откат маловероятен.

4. Для карточки прибыли — добавить sparkline (recharts `<LineChart>` без осей и сетки) в правый нижний угол или под цифрой:
   ```tsx
   <div className="h-10 mt-2">
     <ResponsiveContainer>
       <LineChart data={last7DaysProfit}>
         <Line dataKey="v" stroke="currentColor" strokeWidth={2} dot={false} />
       </LineChart>
     </ResponsiveContainer>
   </div>
   ```
5. Единица валюты — по правилу Задачи 1 (приглушённое `AED`).

### Карточки проектов

1. Структура строки:

- Названия проектов обрезаются через `line-clamp`, не переносятся в 2 строки.

## Откат и альтернативы

# Задача 7. Переработка кнопок действий на странице проекта

## Проблема
На странице проекта 4 кнопки действий стоят вертикально стопкой и занимают половину экрана. «Добавить расход» синяя, остальные — белые пилюли. Декоративные цветные кружки с иконками внутри кнопок — визуальный шум. Плавающий микрофон посреди экрана — непонятный элемент без контекста.

## Цель
Одна главная CTA + компактная сетка вторичных действий. Микрофон — как floating action button, а не inline-элемент.

## Файлы
- `client/src/pages/ProjectDetails.tsx`.
- Возможно: новый компонент `client/src/components/project/ProjectActions.tsx`.

## Реализация

1. **Одна primary CTA сверху — «Добавить расход»**:
   ```tsx
   <Button size="lg" className="w-full">
     <Plus className="w-5 h-5 mr-2" /> Добавить расход
   </Button>
   ```
   Без декоративного кружка внутри.
2. **Вторичные действия — сетка 3 в ряд**:
   ```tsx
   <div className="grid grid-cols-3 gap-2 mt-3">
     <ActionTile icon={FileText} label="Листы реализации" onClick={...} />
     <ActionTile icon={UserPlus}  label="Заказчик"        onClick={...} />
     <ActionTile icon={CheckCircle} label="Завершить"     onClick={...} variant="success" />
   </div>
   ```
   Компонент `ActionTile`:
   ```tsx
   <button className="aspect-square bg-muted/50 hover:bg-muted rounded-xl
                      flex flex-col items-center justify-center gap-1 p-2">
     <Icon className="w-5 h-5 text-primary" strokeWidth={1.75} />
     <span className="text-[11px] font-medium text-center leading-tight">
       {label}
     </span>
   </button>
   ```
3. **Микрофон (голосовой ассистент)** — вынести в **floating action button**:
   ```tsx
   <button className="fixed bottom-24 right-4 w-14 h-14 rounded-full
                      bg-amber-400 shadow-lg flex items-center justify-center z-30
                      active:scale-95 transition">
     <Mic className="w-6 h-6 text-white" />
   </button>
   ```
   `bottom-24` — чтобы не перекрывал таб-бар из Задачи 4.
4. При первом открытии страницы показать tooltip над FAB «Нажмите, чтобы надиктовать расход» (один раз, сохранить в localStorage, что показано).
5. Если голосовой ассистент — не основная функция, можно спрятать его в ActionTile-сетку вместо FAB.

## Критерии приёмки
- Первый экран страницы проекта показывает CTA + сетку действий + заголовок финансовой сводки **без скролла**.
- Микрофон не мешается посреди контента.
- Все действия остались рабочими.

## Откат и альтернативы
- Ветка `feature/design-7-project-actions`.
- Если сетка 3×N смотрится пусто — увеличить до 4 в ряд, сделать иконки поменьше.
- Если FAB сложен в реализации — оставить микрофон как один из ActionTile.

# Задача 8. Консистентность иконок и полировка меню

## Проблема
- Иконки в шапке Dashboard разного цвета и стиля (зелёная, чёрная, синяя, чёрная).
- В меню иконки разной толщины обводки.
- «Подрядчики» и «Сотрудники» используют одну и ту же иконку людей.
- Активный пункт «Главная» в меню оформлен слишком ярко (синяя обводка + фон + текст одновременно).

## Цель
Единый стиль иконок. Смягчённый активный стейт.

## Файлы
- `client/src/components/layout/Header.tsx`.
- `client/src/components/layout/Menu.tsx` (или bottom sheet меню).
- Все места, где импортируются иконки из `lucide-react`.

## Реализация

1. **Унификация иконок**:
   - Везде использовать `lucide-react`.
   - Все иконки — `strokeWidth={1.75}`, размер `w-5 h-5` (или `w-4 h-4` для inline в тексте).
   - Цвет — `text-muted-foreground` по умолчанию, `text-primary` для активного, `text-foreground` для кнопок.
2. **Шапка Dashboard**: убрать 4 разноцветные иконки. Оставить максимум 2: уведомления (колокольчик) + настройки/профиль. Обе — `text-muted-foreground`. Остальное (график, выход) — перенести внутрь профиля или в меню.
3. **Разные иконки для разных сущностей** в меню:
   - Главная — `Home`.
   - Подрядчики — `HardHat` (каска).
   - Заказчики — `Briefcase` или `Building2`.
   - Инструменты — `Wrench`.
   - Персонал — `UserCog`.
   - Сотрудники — `Users`.
   - Настройки — `Settings`.
   - Выход — `LogOut`.
4. **Активный пункт меню** — смягчить:
   ```tsx
   <button className={cn(
     "w-full flex items-center gap-3 px-3 py-2.5 rounded-lg text-sm font-medium",
     active ? "bg-primary/10 text-primary" : "text-foreground hover:bg-muted"
   )}>
     <Icon className={cn("w-5 h-5", active ? "text-primary" : "text-muted-foreground")} />
     {label}
   </button>
   ```
   Без обводки, только светлый фон и цветной текст + иконка.
5. **«Выход»** визуально отделить: `border-t border-border pt-2 mt-2`, цвет текста `text-danger`.

## Критерии приёмки
- Все иконки в приложении одного стиля (обводка одной толщины, один размер).
- В меню нет двух одинаковых иконок для разных сущностей.
- Активный пункт меню выглядит спокойно, не кричит.

## Откат и альтернативы
- Ветка `feature/design-8-icon-polish`.
- Минимальный риск. При проблемах с отдельной иконкой — вернуть прежний вариант точечно.

# Общие принципы (ещё раз)

- **Ничего не ломаем.** Все функциональные сценарии должны работать после каждой задачи идентично.
- **Одна задача — одна ветка — один PR — скриншоты до/после.**
- **Правило отката.** Если задача ломает функциональность или не делает лучше визуально — `git revert`, пробуем альтернативу. Полурабочих решений не оставляем.
- **Не расширяем scope.** Если в процессе замечены другие проблемы — записать в issue, сделать отдельной задачей. Не тащить в текущую ветку.
- **Тестировать на iOS и Android сборках**, не только в браузере.
