# Техническое задание: исправить применение назначенных прав в интерфейсе

## 1. Цель

Сейчас при назначении прав сотруднику через карточку Персонала **галочки сохраняются, но не применяются в интерфейсе**. Сотрудник, которому даны конкретные права, не видит соответствующие разделы и кнопки, либо они ведут не туда, куда должны.

Задача — устранить все баги, при которых **назначенные галочки не превращаются в реально работающие функции**. Любое право, отмеченное в карточке сотрудника, должно полностью работать у этого пользователя: соответствующие разделы видны, кнопки активны, действия выполняются успешно.

## 2. Общие принципы

- Все изменения **корректирующие**: правится неработающая логика связи между галочками прав и UI/функциями. Существующее работающее поведение **не должно сломаться**.
- Перед релизом — обязательное регрессионное тестирование под всеми ролями.
- Все доработки UI — в едином стиле приложения.

## 3. Зафиксированные конкретные баги (на момент написания ТЗ)

Ниже — реальные сценарии, выявленные при тестировании. Они должны быть исправлены и проверены.

### Баг 1. Кнопка «Расходы» ведёт не туда

**Что происходит:** пользователю выданы права на добавление и удаление расходов. Он нажимает в навигации кнопку «Расходы» — открывается раздел «Проекты», а не список расходов.

**Как должно быть:** кнопка «Расходы» открывает раздел «Расходы». Если у пользователя есть право видеть расходы — он видит этот раздел. Если права нет — кнопки в навигации нет вовсе.

### Баг 2. Невозможно добавить расход внутри проекта

**Что происходит:** пользователь заходит в карточку проекта. У него есть право на создание расходов. Но кнопки «Добавить расход» внутри проекта нет, и расход создать невозможно — виден только лист реализации.

**Как должно быть:** при наличии права на создание расхода кнопка «Добавить расход» внутри проекта **видна и активна**, действие выполняется успешно. Должны быть видны и сам список расходов по проекту (если есть право на просмотр), и сами вкладки/разделы по расходам.

### Баг 3. Документы недоступны

**Что происходит:** у пользователя есть права на работу с документами, но раздел документов / вкладка документов внутри проекта отсутствует. Добавлять и удалять документы невозможно.

**Как должно быть:** при наличии прав на просмотр и редактирование документов соответствующая вкладка/раздел видна и работает в полном объёме.

### Баг 4. Нет вкладки «Персонал»

**Что происходит:** пользователю выдан доступ к разделу «Персонал», но соответствующая вкладка/пункт меню в интерфейсе отсутствует.

**Как должно быть:** при наличии права на доступ к «Персоналу» — пункт виден в навигации, раздел открывается и функционирует.

### Общая схема

Эти четыре бага — частные случаи **одной общей проблемы**: галочки в карточке сотрудника сохраняются на сервере, но интерфейс продолжает использовать старую логику (возможно, проверяет роль вместо прав, или использует устаревший кэш, или некоторые экраны вообще не подключены к системе прав).

Поэтому исправлять нужно не каждый баг отдельно, а **системно** — пройтись по всем экранам и убедиться, что каждый из них корректно реагирует на назначенные права.

## 4. Требуемое поведение

### 4.1. Применение прав в навигации

- В нижнем меню, боковом меню и любых других навигационных элементах **видны только те разделы, к которым у пользователя есть права**.
- Если право есть — пункт виден и активен.
- Если права нет — пункт скрыт целиком (не отображается серым).
- Каждая кнопка/пункт ведёт **именно в свой раздел**, а не в другой. Никаких подмен (как сейчас «Расходы» → «Проекты»).

### 4.2. Применение прав внутри экранов

Внутри каждого раздела все кнопки и действия должны **показываться и работать в зависимости от прав**:

- Кнопка «Создать» / «Добавить» — видна, если есть право на создание соответствующей сущности.
- Кнопка «Редактировать» — видна, если есть право на редактирование.
- Кнопка «Удалить» — видна, если есть право на удаление (включая правила вроде «удалять только своё»).
- Списки и таблицы — отображают все объекты, к которым у пользователя есть доступ.
- Вкладки внутри карточки сущности (например, «Расходы», «Документы», «Лист реализации» внутри проекта) — видны в соответствии с правами.

### 4.3. Полный аудит экранов

Команда должна **пройтись по всем экранам приложения** и для каждого проверить:

- Скрыт/показан ли раздел в навигации согласно правам.
- Скрыты/показаны ли все кнопки внутри раздела согласно правам.
- Действительно ли действие выполняется или возвращается отказ при отсутствии права.
- Действительно ли действие выполняется успешно при наличии права.

Минимальный список разделов и сущностей, которые надо проверить (на основании существующей системы):

- Проекты (создание, редактирование, изменение статуса, привязка клиента, назначение мастера)
- Расходы (просмотр, создание с чеком, редактирование, удаление)
- Доходы
- Авансы (сотрудникам и от клиентов)
- Вложения собственника
- Финансовые отчёты (P&L, своды, прибыль/себестоимость)
- Документы по проекту (просмотр, добавление, удаление)
- Листы реализации (просмотр, создание, добавление/удаление фото)
- Персонал
- Сотрудники
- Клиенты, подрядчики
- Инструменты
- Автомобили (база, фотоконтроль, корректировка пробега)
- Аудит-журнал
- Управление сессиями
- Раздел «Права и доступ» (если уже реализован)

Для каждого пункта команда подтверждает: «галочка в карточке сотрудника = реальное поведение в UI».

## 5. Защита со стороны сервера

Параллельно с исправлением UI убедиться, что **сервер тоже корректно проверяет права** на каждом действии. Это страховка на случай, если в UI где-то осталась дыра:

- При попытке выполнить действие без права — сервер возвращает понятный отказ.
- При запросе данных без права — сервер не отдаёт скрытую информацию (например, если у пользователя нет права видеть финансы — он не должен получать их даже через прямой запрос в обход интерфейса).

## 6. Отзывчивость изменений

- После сохранения галочек в карточке сотрудника изменения должны применяться у пользователя **в течение разумного короткого времени** — при следующем действии в приложении, в крайнем случае при следующем входе.
- Если на момент изменения пользователь работает в приложении — он не должен оставаться с устаревшим набором прав надолго.
- В UI карточки сотрудника после сохранения — явное подтверждение «Сохранено и применено».

## 7. Тестирование (приёмка)

Задача считается выполненной, когда успешно проходят следующие сценарии:

### Сценарий A — Расходы

1. Создать тестового сотрудника, выдать только права на просмотр, создание и удаление расходов.
2. Войти под ним: пункт «Расходы» виден в навигации, ведёт в раздел «Расходы» (не в «Проекты»).
3. Раздел «Расходы» открывается, в нём виден список и кнопка «Добавить расход».
4. Добавление расхода с чеком успешно проходит.
5. Удаление своего расхода успешно проходит.
6. Внутри карточки проекта (если у него есть доступ к проекту) — также работает добавление/удаление расходов.
7. Снять права на расходы — пункт исчезает из навигации, кнопок внутри проекта нет.

### Сценарий B — Документы

1. Выдать сотруднику права на просмотр, добавление и удаление документов проекта.
2. Войти под ним: внутри проекта видна вкладка/раздел «Документы».
3. Загрузка документа — работает.
4. Удаление документа — работает.
5. Снять права — вкладка исчезает.

### Сценарий C — Персонал

1. Выдать сотруднику доступ к разделу «Персонал».
2. Войти под ним: пункт «Персонал» виден в навигации.
3. Раздел открывается, отображается список сотрудников.
4. Снять право — пункт исчезает.

### Сценарий D — Сквозная проверка

1. Создать тестового сотрудника с **минимальным** набором прав (например, только просмотр своих проектов).
2. Войти под ним: видны только разрешённые разделы и действия, никаких лишних.
3. Поэтапно добавлять права (по одному) через карточку Персонала.
4. После каждого добавления — соответствующая функция появляется и работает.
5. Поэтапно снимать права.
6. После каждого снятия — соответствующая функция исчезает.

### Сценарий E — Серверная защита

1. Сотрудник без права на просмотр финансов пытается обратиться к данным напрямую (через инструменты разработчика, прямой запрос).
2. Сервер возвращает отказ, данные не утекают.

Перед сдачей команда предоставляет заполненный чек-лист по всем разделам из п. 4.3, где для каждого пункта подтверждено корректное поведение.

## 8. Совместимость

- Текущие учётные записи и их работающие функции **не должны пострадать** — если у пользователя что-то работало до релиза, после релиза должно работать так же или лучше.
- Текущая логика ролей сохраняется как есть, исправляется только связка «галочка → реальное поведение».
- Все ранее реализованные функции (фотоконтроль, листы реализации, отправка чеков в Telegram и т. д.) продолжают работать.

## 9. Что НЕ входит в задачу

- Переработка самой системы ролей и прав — она остаётся, исправляется только её применение в UI и на сервере.
- Изменение дизайна разделов и навигации — только показ/скрытие элементов согласно правам.
- Добавление новых прав, которых сейчас нет в системе.
