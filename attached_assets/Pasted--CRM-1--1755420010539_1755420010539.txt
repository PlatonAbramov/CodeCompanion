ТЗ: Ускорение CRM до ≤1 сек на вкладку (без изменений функционала)
Цель

Добиться времени от клика до готового контента на экране ≤1 сек для всех основных вкладок/страниц при штатной нагрузке. Сохранить текущий функционал 1:1.

Жёсткие ограничения

Не добавлять и не подключать новые сторонние сервисы, интеграции, API.

Не менять бизнес-логику и UI-поведение.

Все оптимизации — «под капотом»: запросы, кэш, статика, рендер, сетевые настройки.

Этап A — Замер и бюджет (без правок логики)
A1. Метрики и пороги

Для каждой страницы зафиксировать:

Время запроса БД (цель: ≤100 мс).

Время подготовки ответа сервером (TTFB цель: ≤200 мс).

Размер HTML+данных (цель: ≤150 КБ на первый ответ).

Время полной отрисовки у клиента (цель: ≤800 мс).

Подготовить «перформанс-паспорт» по страницам: Главная, Проект, Заказчики, Подрядчики, Инструменты, Лист реализации.

A2. Бюджеты

Сетевой бюджет на первый ответ: ≤150 КБ.

Кол-во SQL-запросов на страницу: 1–3 (без N+1).

Время каждого SQL-запроса: ≤50–80 мс; суммарно ≤100 мс.

Изображения в списках: миниатюры ≤25–50 КБ за штуку.

Выход A: таблица метрик «как есть» + целевые бюджеты по каждой странице.

Этап B — Бэкенд-ускорение (без изменения выводимых данных)
B1. Запросы к БД

Убрать N+1: объединить выборки, использовать жадную/точечную подзагрузку данных.

Добавить недостающие индексы по фактическим условиям WHERE/ORDER/JOIN.

Проверить планы выполнения и переписать медленные запросы.

Ограничить поля выборки до реально используемых (без «*» и лишних полей).

B2. Пагинация и лимиты

В списках применять строгие LIMIT (например, 20–50 записей) и чёткий ORDER.

Для тяжёлых подсчётов (суммы/остатки) — использовать предрасчитанные поля/представления без изменения бизнес-логики на UI (результат тот же, быстрее из БД).

B3. Кэш приложения (в памяти сервера)

Горячие неизменные справочники (категории, статусы) — кэш на 5–15 мин.

Заголовок/шапка «финансовой сводки» проекта — краткосрочный кэш 10–30 сек (инвалидировать при изменении записи).

Строго: кэш прозрачен для пользователя, данные всегда актуальны после изменений (инвалидируем точечно).

Выход B: отчёт «до/после» по времени SQL и серверного ответа; описание добавленных индексов/лимитов/кэша.

Этап C — Фронтенд и сеть (без смены дизайна)
C1. Минимизировать передаваемые данные

Убрать из первого ответа лишние поля и вложенные объекты, которые не используются при первичном рендере.

Перенести второстепенные данные на отложенную подзагрузку после отрисовки (те же данные, но позднее).

C2. Статика и изображения

Включить сжатие ответов (Brotli/Gzip на сервере).

Включить долговечные Cache-Control заголовки для неизменной статики.

Для превью фото — отдавать миниатюры; оригиналы — только в модалке по клику.

C3. Рендер

Исключить «блокирующие» операции до первого кадра: инициализировать тяжелые виджеты после рендеринга списка.

Критический контент первым: заголовки, ключевые числа, первые 10–20 строк списка.

Выход C: снижен размер первого ответа, подтверждение включённого сжатия и кеширования, профили «до/после» по FCP/LCP.

Этап D — Регрессия, безопасность и откат
D1. Набор регресc-тестов

Пройти чек-лист всех разделов: визуальные поля совпадают, суммы и статусы те же.

Сравнить случайную выборку страниц «до/после» по содержимому — 1:1.

D2. Постепённый rollout

Включать оптимизации по разделам: сначала «Инструменты», затем «Проекты», далее остальное.

На каждом шаге — контроль метрик и возможность мгновенного отката.

Выход D: акт регресса, журнал развёртываний, чек-лист отката.

Этап E — Приёмка и постоянный контроль
E1. Критерии «сделано»

На каждой целевой странице:

SQL ≤ 100 мс, количество запросов ≤3.

TTFB ≤ 200 мс.

Размер первого ответа ≤150 КБ.

От клика до видимого контента ≤1 сек (среднее по 20 прогонам).

Ни один пользовательский сценарий не изменился.

E2. Мониторинг (внутренний)

Логировать длительные запросы (>100 мс) и страницы (>1 сек).

Еженедельный отчёт: топ-5 «медленных мест», план микрофиксов.

Выход E: финальный отчёт метрик «было/стало» + включённый мониторинг «узких мест».

Чек-лист задач (коротко)

Замеры «как есть» по всем страницам → бюджеты.

SQL-профайлинг: убрать N+1, индексы, лимиты, сузить поля.

Кэш приложения для горячих данных + аккуратная инвалидизация.

Сжатие и кеш статики, миниатюры для превью, отложенная подзагрузка второстепенных данных.

Приоритетный рендер ключевого контента.

Регресс-проверки: данных и UI 1:1.

Постепённый rollout с возможностью отката.

Финальная приёмка по целям и постоянный мониторинг.
