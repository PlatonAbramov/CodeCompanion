ТЗ: Роли и права доступа в CRM
Цель

Разграничить доступ к данным и действиям в CRM по ролям:

Администратор

Прораб

Мастер

Заказчик

Каждая роль видит только то, что нужно, и может делать только разрешённые действия.

Общие правила (для всей системы)

Авторизация и роль

После входа система знает роль пользователя.

Все запросы проверяются на право: «кто?», «к чему?», «что делает?».

Привязка данных к проектам

Прораб и Мастер видят только назначенные им проекты и связанные сущности.

Заказчик видит только свои проекты.

CRUD-ограничения

Разделить права на: Создать / Читать / Обновить / Удалить по каждому модулю.

Удаление критичных сущностей — только при наличии права и без нарушения целостности (например, нельзя удалить проект с платежами без явной проверки).

Аудит

Логировать: кто, когда, что изменил (до/после).

Для финансов — обязательно.

Уведомления

По важным событиям (изменение статуса проекта, добавление расходов, отчётов) — отправлять уведомления тем, кому положено.

Ошибки и запреты

При попытке запрещённого действия — понятное сообщение «Нет прав на действие».

Никаких частичных изменений: либо всё прошло, либо откат.

Поведение интерфейса (UX) по ролям
Администратор

Видит все разделы.

Кнопки «Создать», «Редактировать», «Удалить» доступны везде.

Финансовые формы: авансы, платежи, расходы — доступны.

Экспорт доступен.

Прораб

Главная: только назначенные проекты.

В проекте: можно менять статус проекта, добавлять расходы, загружать документы.

Финансы: только просмотр суммарных показателей проекта (без редактирования сумм договора, без авансов).

Заказчик/подрядчик: только контакты по своим проектам (без изменений).

Инструменты: видит список; может выдать инструмент сотруднику в рамках своих проектов и оформить возврат.

Мастер

Главная: только назначенные проекты.

В проекте: может добавлять свои расходы (с фото чека), просматривать документы.

Инструменты: видит, что у него на руках; может оформить возврат.

Больше ничего не редактирует.

Заказчик

Видит только свои проекты.

Видит стадию/статус, медиаматериалы, документы, помеченные «для заказчика».

Внутренние финансы скрыты.

Только просмотр, без кнопок редактирования.

Правила безопасности данных

Любые списки фильтруются по правам: чужие проекты не попадают в выборку.

Любые действия перед сохранением проверяются: «имеет ли право этот пользователь делать это с этим объектом».

Доступ к файлам (документам) — через проверку роли и принадлежности к проекту.

История изменений хранится с указанием пользователя и времени.

Бизнес-правила (важные мелочи)

Удаление проектов: допускается только Администратору и только если нет незакрытых финансов (или с явным подтверждением).

Расходы:

Прораб видит все расходы в своих проектах, добавляет новые.

Мастер видит только свои расходы и может добавлять новые от своего имени.

Документы:

При загрузке документа внутренняя команда помечает, виден ли он Заказчику.

Заказчик видит только помеченные.

Инструменты:

Статус «у кого находится» меняется только через передачу/возврат.

Мастер не может самолично «выдать» инструмент — только Прораб/Администратор.

Уведомления (минимум)

Изменён статус проекта → Прораб, Мастер (по проекту).

Добавлен расход → Прораб, Администратор.

Передача/возврат инструмента → участники операции, Администратор.

(Позже можно включить уведомления для Заказчика при смене этапа/статуса.)

Проверки и валидации

Суммы и даты — обязательные поля там, где нужно.

Расход — только с категорией и чеком (фото/файл).

Передача инструмента — только реальному пользователю; нельзя выдать занятый инструмент второй раз.

Документы — проверка формата и размера.

Чек-лист реализации (по шагам)

A. Модель ролей

 Добавить роли: Администратор, Прораб, Мастер, Заказчик.

 Привязать роль к пользователю.

 Middleware/проверка прав на каждый запрос.

B. Фильтрация данных

 Проекты: фильтр «только назначенные» для Прораба/Мастера.

 Проекты: «только свои» для Заказчика.

 Подрядчики/контакты: видимость только в контексте своих проектов.

 Инструменты: видимость «у меня» для Мастера.

C. Ограничения действий

 CRUD по каждому модулю согласно матрице.

 Блокировка удаления без прав.

 Блокировка финансовых действий без прав.

 Отдельная проверка на «для заказчика» у документов.

D. Интерфейс

 Скрыть недоступные кнопки/действия по роли.

 Понятные сообщения «Нет прав» при прямом обращении к действиям.

 Отдельные списки: «мои проекты» для Прораба/Мастера/Заказчика.

E. Аудит и уведомления

 Логирование всех изменений критичных данных.

 Базовые уведомления по событиям (см. раздел «Уведомления»).

F. Тест-кейсы (приёмка)

 Войти под каждой ролью и проверить видимость разделов.

 Попробовать запретные действия → получить запрет.

 Прораб видит только «свои» проекты и может добавить расход.

 Мастер видит свои проекты и может добавить только свой расход.

 Заказчик видит только свои проекты и отмеченные документы.

 Инструмент нельзя выдать дважды, возврат меняет статус.

 Экспорт доступен только Администратору.
