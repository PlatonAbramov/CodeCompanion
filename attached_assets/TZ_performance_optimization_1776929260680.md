# Техническое задание: оптимизация производительности приложения

## Цель
Сократить время перехода между страницами и открытия приложения до минимума (цель — переходы без видимого ожидания, первичная загрузка ≤ 1 сек).

---

## Общие правила для всех задач (обязательно к исполнению)

1. **Отдельная ветка и отдельный PR на каждую задачу.** Именование: `feature/perf-<номер>-<короткое-название>`. Смешивать оптимизации в одном PR запрещено — иначе невозможно понять, что именно ускорило или сломало.
2. **Замер ДО и ПОСЛЕ.** Перед началом каждой задачи снять метрики в Chrome DevTools (Network + Performance) на эталонных сценариях (см. раздел «Тестовые сценарии»). Записать в PR-описание. После реализации — снять ещё раз.
3. **Не менять бизнес-логику.** Запрещено: править структуру БД (за исключением задач, где это прямо указано), менять контракты API (формат запросов/ответов), менять роли и права доступа.
4. **Не менять UX.** Все существующие пользовательские сценарии должны работать идентично. Визуальные изменения допустимы только там, где это прямо прописано в задаче (skeleton-заглушки).
5. **Правило отката.** Если задача после реализации:
   - ломает хотя бы один тестовый сценарий,
   - не даёт измеримого прироста скорости,
   - вызывает регресс на продакшне,
   
   разработчик делает `git revert` всей ветки, возвращает код в исходное состояние и пробует альтернативное решение из раздела «Альтернативы» данной задачи. Запрещено оставлять полуработающее решение «до лучших времён».
6. **Прогон тестовых сценариев.** После каждой задачи прогнать полный чек-лист (раздел «Тестовые сценарии»).
7. **Код-ревью.** Минимум один ревью перед мержем в основную ветку. В описании PR — замеры до/после в цифрах.

# Задача 1. Разделение кода по маршрутам (route-based code splitting)

## Проблема
Сейчас все страницы приложения компилируются в один JS-бандл. При первом открытии клиент грузит код всех разделов, даже если пользователь зайдёт только на Dashboard.

## Цель
Разделить бандл по страницам. Каждая страница — отдельный chunk, который грузится только при переходе.

## Файлы
- `client/src/App.tsx` — основной роутер.
- `client/src/pages/*.tsx` — все компоненты страниц.

## Реализация
1. Каждую страницу импортировать через `React.lazy`:
   ```tsx
   import { lazy, Suspense } from 'react';
   const Dashboard = lazy(() => import('./pages/Dashboard'));
   const ProjectDetails = lazy(() => import('./pages/ProjectDetails'));
   // ...все остальные страницы
   ```
2. Обернуть роуты Wouter в `<Suspense>` с fallback-компонентом (на первом этапе — минимальный спиннер, в Задаче 6 заменим на skeleton).
3. Если у страницы named export — использовать:
   ```tsx
   const Page = lazy(() => import('./pages/Page').then(m => ({ default: m.Page })));
   ```
4. Проверить в `dist/` после сборки `vite build`: должны появиться отдельные chunk-файлы для каждой страницы.

## Критерии приёмки
- Размер main chunk (index-*.js) уменьшается минимум на 30%.
- В Network-логе при переходе на новую страницу виден отдельный JS-chunk.
- Все страницы открываются без ошибок консоли.
- Все тестовые сценарии проходят.

## Откат и альтернативы
- Ветка `feature/perf-1-code-splitting`.
- Если один из lazy-чанков падает (ошибка импорта, циклические зависимости) — `git revert`, разобраться с причиной и попробовать снова.
- Альтернатива: `manualChunks` в vite.config.ts — группировка по доменам (admin, common, public) вместо по страницам.

# Задача 2. Оптимизация TanStack Query (устранение дублей и перезапросов)

## Проблема
Согласно `performance_analysis.md`: `/api/auth/me` вызывается 3–4 раза подряд, `/api/projects/{id}/financial-summary` — до 6 раз. При каждом переключении вкладки данные запрашиваются заново, пользователь видит спиннер.

## Цель
Настроить TanStack Query так, чтобы: а) не было дублирующих запросов, б) при возврате на ранее посещённую страницу данные показывались мгновенно из кэша, в) свежие данные подтягивались в фоне.

## Файлы
- `client/src/lib/queryClient.ts` (или место создания `QueryClient`).
- Все компоненты с `useQuery` в `client/src/pages/*` и `client/src/components/*`.

## Реализация
1. В `QueryClient` выставить глобальные дефолты:
   ```ts
   new QueryClient({
     defaultOptions: {
       queries: {
         staleTime: 30_000,
         gcTime: 5 * 60_000,
         refetchOnWindowFocus: false,
         refetchOnMount: false,
         retry: 1,
         placeholderData: (prev) => prev, // аналог keepPreviousData
       },
       mutations: { retry: 0 },
     },
   });
   ```
2. Точечные staleTime по типу данных:
   - `['auth', 'me']` — `staleTime: Infinity` (инвалидировать только при login/logout).
   - Списки (projects, clients, contractors, users) — `staleTime: 60_000`.
   - Финансовые сводки — `staleTime: 10_000`.
3. Пройти по всем `useQuery` и проверить, что `queryKey` одинаковый там, где данные одинаковые. Пример правильного ключа: `['projects', projectId, 'expenses']`. Если в разных компонентах один и тот же запрос имеет разные ключи — это и есть причина дублирования.
4. После мутаций (создать проект, добавить расход и т. д.) — точечная инвалидация:
   ```ts
   onSuccess: () => {
     queryClient.invalidateQueries({ queryKey: ['projects'] });
     queryClient.invalidateQueries({ queryKey: ['projects', projectId, 'financial-summary'] });
   }
   ```
5. Запретить автоматический refetch при монтировании компонента (`refetchOnMount: false`). Данные будут показаны из кэша, свежие подтянутся только если `staleTime` истёк.

## Критерии приёмки
- При повторном открытии уже посещённой страницы — нет спиннера, данные показываются мгновенно.
- В Network Log при переходах нет дублей одного и того же эндпоинта в пределах 1 секунды.
- `/api/auth/me` вызывается один раз за сессию.
- Данные, изменённые через мутации, корректно обновляются на всех страницах.

## Откат и альтернативы
- Ветка `feature/perf-2-query-optimization`.
- Если после оптимизации где-то данные «залипают» (пользователь добавил расход — не видит его) — это значит пропущена инвалидация. Починить локально.
- Если чинить дольше дня — `git revert` и реализовать задачу инкрементально, по одной странице за раз.
- Альтернатива: полностью ручное управление через `queryClient.setQueryData` после мутаций (без invalidate).

# Задача 3. Gzip/Brotli-сжатие HTTP-ответов

## Проблема
Express сейчас отдаёт JSON-ответы без сжатия. Ответы со списками проектов, финансовыми сводками, списками расходов — в 5–10 раз больше, чем могли бы быть. На мобильном интернете это критично.

## Цель
Включить сжатие всех текстовых ответов (JSON, HTML, JS, CSS).

## Файлы
- `server/index.ts` — до регистрации роутов.
- `package.json` — добавить зависимость.

## Реализация
1. Установка:
   ```
   npm install compression
   npm install -D @types/compression
   ```
2. В `server/index.ts` **до всех `app.use(router)`**:
   ```ts
   import compression from 'compression';
   app.use(compression({
     threshold: 1024,
     filter: (req, res) => {
       if (req.headers['x-no-compression']) return false;
       return compression.filter(req, res);
     },
   }));
   ```
3. Убедиться, что middleware зарегистрировано **раньше** статики и API-роутов.

## Критерии приёмки
- В заголовках ответов > 1 КБ присутствует `Content-Encoding: gzip`.
- Размер передаваемых JSON в DevTools → Network уменьшается в 3–10 раз (колонка Size vs «размер в памяти»).
- Все API-эндпоинты и загрузка страниц работают корректно.
- Загрузка файлов (multipart, изображения) не сломалась — `compression` по умолчанию не трогает бинарные типы.

## Откат и альтернативы
- Ветка `feature/perf-3-compression`.
- Если какой-то эндпоинт возвращает некорректные данные — проверить, не кастует ли он `Content-Type` вручную. При необходимости добавить исключение через фильтр.
- Если ломается всё — `git revert`.
- Альтернатива: настроить сжатие на уровне обратного прокси (Nginx/Cloudflare, см. Задачу 8) — тогда `compression` не нужен.

# Задача 4. Устранение cold start у Neon Postgres

## Проблема
Neon (serverless Postgres) засыпает при простое. Первый запрос после паузы 5+ минут занимает 300–800 мс только на «прогрев».

## Цель
Держать подключение к БД всегда прогретым. Первый запрос после простоя — так же быстр, как последующие.

## Файлы
- `server/db.ts`.
- Переменные окружения (`.env`, настройки продакшн-окружения).

## Реализация
1. Проверить `DATABASE_URL`. В Neon dashboard взять **pooled connection string** (содержит `-pooler` в имени хоста). Использовать именно его. Non-pooled оставить только для миграций.
2. В `server/db.ts` использовать `@neondatabase/serverless` (уже установлен) в режиме Pool.
3. Добавить keep-alive при старте сервера:
   ```ts
   const KEEP_ALIVE_INTERVAL = 4 * 60_000;
   const keepAliveTimer = setInterval(() => {
     db.execute(sql`SELECT 1`).catch(err => console.warn('keepalive failed', err));
   }, KEEP_ALIVE_INTERVAL);
   
   process.on('SIGTERM', () => clearInterval(keepAliveTimer));
   process.on('SIGINT', () => clearInterval(keepAliveTimer));
   ```
4. Проверить в логах Neon, что количество открытых коннектов не растёт бесконтрольно.

## Критерии приёмки
- Первый API-запрос к БД после 5+ минут простоя — TTFB ≤ 200 мс.
- Счётчик соединений в Neon dashboard стабилен.
- Graceful shutdown: при остановке сервера интервал очищается.

## Откат и альтернативы
- Ветка `feature/perf-4-neon-keepalive`.
- Если keep-alive создаёт лишние ошибки в логах — увеличить интервал до 6–8 минут.
- Альтернатива: на платном тарифе Neon отключить Scale to Zero.
- Если всё плохо — `git revert`.

# Задача 5. Оптимизация изображений (миниатюры + lazy loading)

## Проблема
Фото из Листов реализации, чеки, аватары загружаются в оригинальном разрешении (часто 2–5 МБ). Страница с галереей из 20 фото тянет 40–100 МБ.

## Цель
Для списков и галерей показывать миниатюры 25–100 КБ. Оригинал — только по запросу (клик «Открыть в полном размере» / «Скачать»).

## Файлы
- `server/objectStorage.ts` — загрузка и выдача URL.
- `server/routes.ts` — эндпоинты, возвращающие ссылки на фото.
- `client/src/components/*` — компоненты галерей и списков с `<img>`.
- Новый файл: `server/imageProcessor.ts`.

## Реализация
1. Установка:
   ```
   npm install sharp
   ```
2. При загрузке фото в Implementation Sheet / как чек / как документ-изображение — генерировать 3 версии через sharp:
   - `thumb` — 256×256, WebP, quality 75.
   - `medium` — 1024px по длинной стороне, WebP, quality 80.
   - `original` — как есть.
3. Хранить в GCS по структуре:
   ```
   implementation-photos/{itemId}/{photoId}/thumb.webp
   implementation-photos/{itemId}/{photoId}/medium.webp
   implementation-photos/{itemId}/{photoId}/original.{ext}
   ```
4. В БД в таблице `implementation_photos` добавить поля `thumbUrl`, `mediumUrl` (миграция). Оставить существующее поле URL как `originalUrl` или аналогичное.
5. API возвращает все три URL:
   ```ts
   { id, thumbUrl, mediumUrl, originalUrl, ... }
   ```
6. Во фронтенде:
   - Список/галерея — `src={photo.thumbUrl}`.
   - Фуллскрин-вьювер — `src={photo.mediumUrl}`, кнопка «Скачать оригинал» — `originalUrl`.
   - Везде добавить `loading="lazy"` и `decoding="async"`.
7. Для уже загруженных старых фото, у которых нет миниатюр:
   - Скрипт batch-обработки `scripts/backfill-thumbnails.ts`, который идёт по всем записям без `thumbUrl`, генерирует и обновляет БД.
   - Fallback во фронтенде: если `thumbUrl` пустой — использовать `originalUrl`.

## Критерии приёмки
- Страница Листа реализации с 20 фото грузится < 2 МБ (ранее — > 20 МБ).
- Первый экран галереи отрисовывается < 500 мс.
- Клик по фото открывает `medium`, кнопка «скачать» даёт оригинал.
- Старые фото без миниатюр продолжают отображаться (fallback).

## Откат и альтернативы
- Ветка `feature/perf-5-image-optimization`.
- Миграция БД должна быть **обратно совместимой**: добавление новых полей, не удаление старых. Если что-то пойдёт не так — удалить только новые колонки.
- Если sharp не устанавливается на текущий рантайм — использовать облачную обработку (Cloudflare Images / Imgproxy) или Google Cloud Storage resize-on-demand.
- Если всё плохо — `git revert` и полный откат миграции.

# Задача 6. Skeleton-заглушки вместо спиннеров

## Проблема
При загрузке страницы пользователь видит пустой экран или крутящийся спиннер. Это субъективно воспринимается как «приложение тормозит», даже если фактически загрузка занимает 300 мс.

## Цель
Вместо спиннера показывать «каркас» страницы (серые прямоугольники на местах контента). UI ощущается мгновенно отзывчивым.

## Файлы
- `client/src/components/ui/skeleton.tsx` (shadcn/ui, должен уже существовать).
- `client/src/pages/*.tsx`.
- Новые файлы: `client/src/components/skeletons/*.tsx` — по одному на ключевую страницу.

## Реализация
1. Для каждой основной страницы создать skeleton-компонент, повторяющий её сетку:
   ```tsx
   export function ProjectsSkeleton() {
     return (
       <div className="grid gap-4">
         {Array.from({ length: 6 }).map((_, i) => (
           <Skeleton key={i} className="h-24 w-full" />
         ))}
       </div>
     );
   }
   ```
2. Использовать в двух местах:
   - Как `fallback` в `<Suspense>` из Задачи 1.
   - В `isLoading` каждого `useQuery`:
     ```tsx
     if (isLoading && !data) return <ProjectsSkeleton />;
     ```
3. Не показывать skeleton, если уже есть `placeholderData` из предыдущего запроса (из Задачи 2).
4. Скелетон должен быть по структуре **точно таким же**, как финальная сетка — чтобы не было «прыжка» контента при подмене.

## Критерии приёмки
- При переходах между страницами не появляется крутящийся спиннер — только skeleton или сразу данные.
- Нет layout shift при подмене skeleton на данные (проверить в Lighthouse → CLS).

## Откат и альтернативы
- Ветка `feature/perf-6-skeletons`.
- Чисто визуальная задача, риск минимальный. Если где-то skeleton ломает вёрстку — исправить точечно.

# Задача 7. Prefetch данных на hover/viewport

## Проблема
При клике на карточку проекта сначала идёт запрос, потом рендер. Если бы данные начали грузиться заранее (при наведении мыши или когда карточка появляется на экране) — переход был бы мгновенным.

## Цель
Prefetch данных до фактического клика пользователя.

## Файлы
- `client/src/components/ProjectCard.tsx` и аналогичные карточки-ссылки.
- `client/src/pages/Dashboard.tsx`, `client/src/pages/Projects.tsx`.

## Реализация
1. Для десктопа — prefetch на `onMouseEnter` / `onFocus`:
   ```tsx
   const queryClient = useQueryClient();
   const handleHover = () => {
     queryClient.prefetchQuery({
       queryKey: ['projects', project.id, 'financial-summary'],
       queryFn: () => apiFetch(`/api/projects/${project.id}/financial-summary`),
       staleTime: 10_000,
     });
   };
   return <Card onMouseEnter={handleHover} onFocus={handleHover}>...</Card>;
   ```
2. Для мобильных (основная платформа — Capacitor) — через `IntersectionObserver`: как только карточка попадает в viewport, prefetch.
3. Ограничить prefetch:
   - Не более 5 параллельных prefetch одновременно.
   - Prefetch только самых критичных эндпоинтов (финансовая сводка, список расходов), не всё подряд.
   - Не prefetch-ить, если пользователь на медленном соединении (`navigator.connection?.effectiveType === '2g'`).

## Критерии приёмки
- При переходе на ранее не открытую страницу проекта — в 80% случаев нет спиннера.
- Серверная нагрузка не растёт более чем на 15% (проверить по логам).

## Откат и альтернативы
- Ветка `feature/perf-7-prefetch`.
- Если сервер не справляется с нагрузкой — ограничить prefetch только первыми 3–5 видимыми карточками.
- Если всё плохо — `git revert`.

# Задача 8. HTTP/2 + CDN (Cloudflare)

## Проблема
Если сервер отдаёт статику напрямую — каждое соединение проходит через TLS-handshake, без edge-кэширования, без HTTP/2-мультиплексирования. На географически удалённых клиентах RTT 200+ мс на каждый запрос.

## Цель
Поставить Cloudflare перед приложением. Статика отдаётся с edge-серверов, HTTP/2/3, сжатие на уровне CDN.

## Реализация (DevOps)
1. Зарегистрировать домен в Cloudflare (бесплатный тариф).
2. Перевести DNS на Cloudflare nameservers.
3. Для записи домена включить proxy (оранжевое облако).
4. Cache Rules:
   - `/assets/*`, `*.js`, `*.css`, `*.woff*`, `*.png`, `*.webp` — Cache Everything, Edge TTL = 1 month.
   - `/api/*` — Bypass cache.
5. Включить: Brotli, HTTP/2, HTTP/3 (QUIC), 0-RTT.
6. В Express на раздаче статики выставить заголовки:
   ```ts
   app.use('/assets', express.static('dist/public/assets', {
     maxAge: '1y',
     immutable: true,
   }));
   ```
7. SSL mode в Cloudflare — Full (strict), чтобы не ломать https.

## Критерии приёмки
- В Response Headers для статики есть `cf-cache-status: HIT`.
- TTFB для статики в удалённых регионах < 50 мс.
- API продолжает работать без изменений.

## Откат

# Общие принципы (ещё раз)

- **Не ломаем существующую структуру.** Каждая задача — улучшение поверх того, что есть. Если видите соблазн «заодно переписать X» — не делайте этого.
- **Правило отката.** Всё, что не даёт измеримого результата или ломает функциональность, — немедленно откатывается через `git revert`.
- **Измеряем.** В каждом PR — цифры до/после (TTFB, размер бандла, время до интерактивности).
- **Работаем инкрементально.** Одна задача — одна ветка — один PR — один замер.
