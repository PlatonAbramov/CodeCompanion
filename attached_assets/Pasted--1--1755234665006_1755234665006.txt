ТЗ: Простая и надёжная аутентификация «вход по логину/паролю»
1) Цель и результат

Цель: дать доступ к приложению только авторизованным пользователям через форму входа.
Результат: продакшн-готовый модуль входа/выхода с хранением пользователей в БД, управлением пользователями через админку/API, сессиями и базовой безопасностью.

2) Функциональные требования
2.1 Вход

Поля: login (уникальный), password.

Опционально: простая MFA TOTP (вкл/выкл на пользователя) — по возможности.

Защита от перебора: rate limiting + временные блокировки.

2.2 Выход

Кнопка Logout: инвалидировать текущую сессию/refresh-токен.

2.3 Управление пользователями (только админ)

Создать пользователя: login, временный пароль, имя (опц.), роль (user/admin).

Сбросить пароль пользователю (генерация временного).

Включить/выключить MFA.

Заблокировать/разблокировать пользователя.

Просмотр активных сессий пользователя и принудительный выход «из всех устройств».

2.4 Сессии

Модель access + refresh:

access (JWT, ~15 минут).

refresh (HTTP-only Secure cookie, ~30 дней) — привязать к устройству.

Возможность «Logout all» для конкретного пользователя (админ/сам пользователь).

3) Нефункциональные требования
3.1 Безопасность паролей

Хранение только хэша: Argon2id (предпочтительно) или bcrypt (cost ≥ 12).

Индивидуальная соль + pepper (секрет в ENV).

Сравнение хэшей — константным временем.

3.2 База данных (PostgreSQL 14+)

Минимальная схема:

users

id (UUID, PK)

login (text, unique, not null)

password_hash (text, not null)

password_algo (text)

role (enum: user/admin, default user)

is_blocked (bool, default false)

mfa_enabled (bool, default false)

created_at, updated_at, deleted_at (soft delete, опц.)

sessions

id (UUID, PK)

user_id (FK)

refresh_token_hash (text)

user_agent (text), device_label (text), ip_hash (text)

created_at, last_used_at, expires_at, revoked_at (nullable)

login_attempts

id, user_id (nullable), ip_hash, result (success/failed/locked), created_at

mfa_totp (если включаем MFA)

user_id (PK, FK)

secret_encrypted (text)

recovery_codes (jsonb, опц.)

created_at, updated_at

Индексы: уникальный на users.login; по (user_id, revoked_at) в sessions; по created_at/ip_hash в login_attempts.

3.3 Инфраструктура/деплой

Разделить staging и production, разные секреты.

HTTPS, HSTS. Cookie: HttpOnly, Secure, SameSite=Lax/Strict.

Секреты только в ENV (Replit Secrets / CI).

Для Replit: Deploy → Web Service, проставить переменные окружения.

3.4 Логи и аудит (минимум)

Логировать: успешные/неуспешные входы, блокировки, админ-изменения (создание/сброс/блокировка).

Маскировать логины/IP (хэш/частичное скрытие).

3.5 Резервирование

Ежедневный бэкап БД, проверка восстановления раз в месяц.

4) API (минимальный набор)

POST /auth/login → 200
Вход: login, password, (mfa_code если включено).
Выход: access (JWT) + установка refresh (cookie) + профиль.

POST /auth/refresh → 200
Обновляет access по refresh (cookie); проверка fingerprint.

POST /auth/logout → 204
Инвалидирует текущую сессию (refresh).

POST /auth/logout-all → 204
Инвалидирует все сессии пользователя.

GET /me → 200
Профиль (login, роль); список активных устройств (кратко).

Админ (только admin):

POST /admin/users → создать пользователя (login, роль, временный пароль).

POST /admin/users/{id}/reset-password → выдать временный пароль.

POST /admin/users/{id}/block / unblock.

POST /admin/users/{id}/logout-all.

PATCH /admin/users/{id} → изменить роль, имя, mfa_enabled.

GET /admin/users → поиск/фильтр (по login/статусу).

Единый формат ошибок: { error: { code, message, details } }.

5) UI потоки

Вход:

Пользователь вводит login/пароль.

Если включена MFA → поле кода.

При успехе — редирект в приложение; при 5–10 неудачных попытках подряд — временная блокировка (например, 15 минут).

Выход: кнопка «Выйти» → серверная инвалидизация refresh.

Админка:

Список пользователей, поиск по login.

Кнопки: «Создать», «Сброс пароля», «Блокировать», «Разблокировать», «Вывести из всех устройств».

Переключатель MFA (если используем).

6) Защита и ограничения

Rate limiting:

/auth/login: 5–10 попыток/мин на IP и/или login; прогрессивная задержка.

Блокировка пользователя при 10–15 неудачных входах за 15 мин (конфигурируемо).

Анти-enumeration: одинаковые ответы/задержки для «неверный login/пароль».

CSRF: если используем cookie для refresh, защищать state-изменяющие запросы CSRF-токеном (или держать авторизацию только в header).

CORS: ограничить доверенные origin.

Security headers: CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy.

7) Переменные окружения (пример)

APP_ENV (staging/production)

DB_URL (Postgres)

REDIS_URL (rate limit, сессии blacklist)

JWT_SECRET, JWT_ACCESS_TTL=15m, JWT_REFRESH_TTL=30d

PASSWORD_PEPPER

RATE_LIMIT_LOGIN_PER_MIN, LOCK_THRESHOLD, LOCK_TTL_MIN

ALLOWED_ORIGINS

CSP (при необходимости)

8) Тестирование и приёмка (минимум)

Юнит-тесты

Хэширование/сравнение паролей (argon2id/bcrypt).

Логику блокировки и лимитов.

Инвалидизацию сессий (logout/logout-all).

Интеграционные

Вход → доступ к защищённым эндпоинтам → refresh → logout.

Блокировка после серии неудачных входов.

Админ-действия: создать/сбросить/блокировать/выход-везде.

Безопасность

Cookie-флаги, CORS, headers.

Равные ответы при неверных логин/пароль.

Проверка, что паролей в логах/ответах нет.

Критерии приёмки (must-have)

Рабочие вход/выход/refresh; доступ к API закрыт без токена.

Админ может создавать/блокировать/сбрасывать пароли.

Защита от перебора (rate limit + блокировка).

Пароли в БД только в виде хэшей (соль + pepper).

Бэкапы настроены, env-секреты вынесены.
