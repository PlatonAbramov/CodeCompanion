# Техническое задание: полная сквозная проверка работы прав по всему приложению

## 1. Цель

В приложении регулярно встречаются ситуации, когда **галочка права отмечена у сотрудника, но функция не работает** (или наоборот: галочка снята, а доступ остаётся). Последний пример: сотруднику выдан доступ к разделу «Персонал», галочка стоит — но при попытке открыть раздел система показывает «Нет доступа».

Это уже не локальный баг конкретного экрана, а **системная проблема связи** между галочками прав и фактическим поведением приложения.

Задача — провести **полную сквозную проверку всех прав по всему приложению** и устранить любые рассинхроны. Главное правило, которое должно работать без исключений:

> **Если галочка стоит — функция работает. Если галочка снята — функция не работает.**
> Это правило действует для пользователя любой роли (Админ, Директор, Мастер, Прораб, Рабочий, Клиент) и для любого права.

## 2. Общие принципы

- Все изменения **корректирующие**: правится логика проверки и применения прав там, где она сейчас не работает корректно. Существующее работающее поведение **не должно сломаться**.
- Перед релизом — обязательное регрессионное тестирование под каждой ролью.
- Все доработки UI — в едином стиле приложения.

## 3. Суть проблемы

Симптомы, которые наблюдаются сейчас:

- Право выдано → раздел или действие недоступны.
- Право снято → раздел или действие остаются доступны.
- Право выдано → раздел открывается, но кнопки внутри неактивны или отсутствуют.
- Право выдано → нажатие на кнопку ведёт не туда (например, «Расходы» открывает «Проекты»).
- Право выдано → сервер возвращает «Нет доступа» при попытке выполнить действие, на которое права формально есть.

Все эти симптомы — частные случаи **одной общей проблемы**: между галочкой в карточке сотрудника и фактическим поведением приложения нарушена связь. Где-то проверяется не то право, где-то проверяется роль вместо права, где-то используется устаревший кэш, где-то проверка вообще не подключена.

Поэтому исправлять нужно **системно, а не точечно**.

## 4. Требуемое поведение

### 4.1. Главное правило

Для **каждого** права в системе и для **каждой** роли:

- **Галочка стоит** → соответствующий раздел/кнопка/действие видны и работают в полном объёме у пользователя, которому выдано это право.
- **Галочка снята** → соответствующий раздел/кнопка/действие недоступны: либо скрыты, либо неактивны, и сервер отказывает при попытке выполнить запрещённое действие напрямую.

Это правило не имеет исключений и не зависит от роли. Назначены права Мастеру, Прорабу, Рабочему, Клиенту, Админу — поведение одинаково: галочка управляет реальным поведением.

### 4.2. Согласованность между UI и сервером

UI и сервер должны **проверять одно и то же право**:

- Если UI показывает кнопку — сервер должен принимать соответствующий запрос.
- Если UI скрывает кнопку — сервер должен отказывать в соответствующем запросе.
- Не должно быть ситуаций «кнопка есть, а действие выдаёт ошибку» или «кнопки нет, но действие можно выполнить через прямой запрос».

### 4.3. Полный аудит всех прав и всех экранов

Команда должна провести **сквозной аудит** и для каждого права в системе зафиксировать:

1. Какие элементы интерфейса оно контролирует (пункты меню, вкладки, кнопки, поля).
2. Какие действия на сервере оно контролирует.
3. Что должно происходить при наличии права.
4. Что должно происходить при отсутствии права.

Минимальный список разделов и сущностей, которые надо проверить:

- Проекты (создание, редактирование, изменение статуса, привязка клиента, назначение мастера, просмотр своих/всех/назначенных)
- Расходы (просмотр, создание с чеком, редактирование, удаление; в навигации и внутри проекта)
- Доходы
- Авансы (сотрудникам и от клиентов)
- Вложения собственника
- Финансовые отчёты (P&L, своды, прибыль, себестоимость)
- Документы по проекту (просмотр, добавление, удаление)
- Листы реализации (просмотр, создание, добавление и удаление фото — включая правило «удалять только своё»)
- Персонал (просмотр, создание, редактирование, привязка учётной записи, назначение ролей и прав)
- Сотрудники / учётные записи
- Клиенты, подрядчики
- Инструменты
- Автомобили — база (создание, редактирование, архивирование, привязка водителя)
- Автомобили — фотоконтроль (прохождение, просмотр своих/всех, корректировка пробега)
- Аудит-журнал
- Управление сессиями
- Раздел «Права и доступ»

Для каждого пункта команда подтверждает: «галочка в карточке сотрудника = реальное поведение и в UI, и на сервере».

### 4.4. Согласованность между «Персоналом» и учётной записью

- Галочки прав, отмеченные в карточке Персонала, должны однозначно соответствовать реальному набору прав привязанной учётной записи. Без расхождений.
- При открытии карточки Персонала галочки отображают **актуальное реальное состояние** прав привязанной учётной записи.
- Источник правды по правам — единственный. Двойного хранения с риском расхождения быть не должно.
- При изменении галочек — изменения применяются у пользователя в течение разумного короткого времени (при следующем действии в приложении или, в крайнем случае, при следующем входе).

## 5. Защита со стороны сервера

Все проверки прав выполняются **и на клиенте, и на сервере**. Клиентские проверки — для удобства интерфейса. Серверные проверки — основная защита.

При попытке выполнить действие без права (через прямой запрос в обход интерфейса) — сервер отказывает с понятным сообщением. Без скрытой утечки данных, к которым у пользователя нет доступа.

## 6. Сценарии тестирования (приёмка)

Задача считается выполненной, когда успешно проходят следующие сценарии. По каждому — фиксируется результат в чек-листе.

### Сценарий 1. Минимум прав → нет доступа

1. Создать тестового сотрудника без всяких прав.
2. Войти под ним: в навигации видно только то, что не требует прав (профиль, выход).
3. Прямые запросы к закрытым разделам — сервер отказывает.

### Сценарий 2. Поэтапная выдача прав

1. Этому же сотруднику поочерёдно выдавать по одному праву за раз.
2. После каждой выдачи — соответствующий раздел/кнопка/действие появляется и работает.
3. Никакие лишние разделы или возможности при этом не появляются.

### Сценарий 3. Поэтапное снятие прав

1. Поочерёдно снимать каждое выданное право.
2. После каждого снятия — соответствующий раздел/кнопка/действие исчезает.
3. Прямой запрос на снятое действие — сервер отказывает.

### Сценарий 4. Конкретные ранее зафиксированные баги

Эти случаи отдельно проверяются на тестовом стенде:

- Право на расходы выдано → пункт «Расходы» в навигации виден и ведёт в раздел «Расходы», а **не** в «Проекты». Внутри проекта виден список расходов и кнопка «Добавить расход», добавление и удаление работают.
- Права на документы выданы → внутри проекта видна вкладка «Документы», добавление и удаление работают.
- Право на «Персонал» выдано → пункт «Персонал» виден в навигации, раздел открывается **без** сообщения «Нет доступа».
- Все вышеперечисленные пункты — отдельно проверяются и при снятии прав: исчезают и недоступны.

### Сценарий 5. Сквозная проверка всех ролей

1. Под каждой ролью (Админ, Директор, Мастер, Прораб, Рабочий, Клиент) с дефолтным набором прав войти в систему.
2. Убедиться, что видимый интерфейс точно соответствует матрице прав этой роли.
3. Все ключевые действия, разрешённые роли, успешно выполняются.
4. Все ключевые действия, запрещённые роли, недоступны и в UI, и на сервере.

### Сценарий 6. Согласованность Персонал ↔ Учётная запись

1. Открыть карточку Персонала — посмотреть отмеченные галочки.
2. Войти под привязанной учёткой — все отмеченные галочки реально работают.
3. Изменить галочки в карточке Персонала.
4. Реальные права у пользователя меняются в течение разумного времени.
5. Открыть карточку Персонала ещё раз — галочки точно соответствуют тому, что только что изменили (никакого «отката» к старому состоянию).

### Сценарий 7. Серверная защита

1. Под пользователем с ограниченными правами попытаться вызвать запрещённые действия напрямую (через инструменты разработчика).
2. Сервер на каждое такое обращение отказывает.
3. Никакие закрытые данные не отдаются.

Перед сдачей команда предоставляет заполненный чек-лист со всеми правами и пройденными сценариями. На каждое право — отметка: «галочка работает корректно (в UI и на сервере)».

## 7. Совместимость

- Текущие учётные записи и их рабочие функции **не должны пострадать**.
- Текущая логика ролей сохраняется как есть, исправляется только связка «галочка → поведение».
- Все ранее реализованные функции продолжают работать.

## 8. Что НЕ входит в задачу

- Переработка самой системы ролей и прав.
- Добавление новых прав, которых сейчас нет.
- Изменение дизайна разделов и навигации — только корректное показ/скрытие элементов согласно правам.
