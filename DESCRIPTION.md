# Pag CRM — описание системы

Внутренняя CRM-система для строительной и HVAC-компании Platon Abramov Group.
Веб-приложение мобильного дизайна (адаптировано под смартфоны и планшеты), помогает
вести проекты, контролировать финансы, отслеживать работы по сметам, управлять
персоналом, транспортом и инструментом.

---

## 1. Кто работает в системе (роли)

В системе пять ролей. Каждой можно гибко настраивать права на странице
«Права и доступ» — права применяются ко всем эндпоинтам бэкенда.

| Роль | Кто это | Что обычно может |
|---|---|---|
| **Админ** | Платон / системный админ | Всё. Создание пользователей, гибкие права, админ-панель, журнал действий, обнуление паролей. |
| **Директор** | Руководитель | Полный доступ к проектам, финансам, отчётам, персоналу, клиентам, подрядчикам. |
| **Мастер** | Прораб / бригадир | Свои расходы, авансы, фотоконтроль реализации, документы своих проектов. |
| **Рабочий** | Исполнитель на объекте | Список активных проектов (без финансов), просмотр листов реализации, загрузка фото с объекта. Если назначена роль «Водитель» — доступ к разделу «Транспорт». |
| **Клиент** | Заказчик | Только чтение по своим проектам: листы реализации, фото, прогресс. Без правок и финансовых данных. |

Регистрация снаружи закрыта — учётки заводит только админ. Пароли хранятся
в зашифрованном виде (bcrypt). Сессии хранятся в БД, можно принудительно
разлогинить пользователя из админки.

## 2. Основные разделы

### 2.1 Проекты
- Создание и редактирование проектов, бюджет, сроки, статусы, архивирование.
- Привязка клиента и подрядчиков.
- Назначение исполнителей (мастеров и рабочих).
- Карточка проекта показывает: финансовую сводку, расходы, доходы, авансы клиента,
  авансы сотрудникам, документы, фото, листы реализации.
- Финансовая сводка считается в реальном времени (P&L: доходы − расходы − авансы).
- Архив проектов — отдельная страница для завершённых работ.

### 2.2 Финансы

**Расходы (Expenses)**
- Быстрая мобильная форма ввода с обязательным фото чека/счёта.
- Категории расходов и группировка.
- Привязка к проекту и подрядчику.
- Голосовой ввод (hold-to-talk) — кнопка зажата, диктуешь, отпускаешь —
  система распознаёт сумму, проект и описание.

**Доходы (Revenues)**
- Поступления по проектам с подтверждающими документами.
- Сводка по проектам и общая выручка.

**Авансы клиентов (Customer Advances)**
- Предоплаты заказчиков по проекту, история и статусы.

**Авансы сотрудникам (Personnel Advances)**
- Наличные/безналичные выдачи персоналу под проект.
- Возможность отменить аванс с указанием причины (с журналом).
- Сводка по человеку и по проекту.

**Инвестиции владельца (Owner Investments)**
- Учёт собственных вложений в проект для корректного P&L.

**Финансовый обзор**
- Общая выручка, расходы, прибыль.
- Финансовая сводка по каждому проекту.
- Отдельная страница «Аналитика» с разбивками.

### 2.3 Персонал (HR)
- Карточка сотрудника: ФИО, контакты, дата рождения, Emirates ID
  (с датами выдачи и истечения), специализация, дата начала работы, оклад,
  фото профиля.
- Документы сотрудника (виза, ВУ и т. д.) с датами истечения и подсветкой
  просроченных.
- Авансы сотрудника с балансом.
- Журнал изменений роли «Водитель».
- **Связь с учётной записью** — админ может прямо из карточки персонала
  создать учётку (логин/пароль/роль) или привязать существующую. Через
  привязку работают индивидуальные права на странице «Права и доступ».
- Роль «Водитель» — флаг, дающий доступ к разделу «Транспорт».
  Снять роль нельзя, если за человеком закреплён активный авто.

### 2.4 Клиенты
- База клиентов (физлица и компании), контакты, реквизиты.
- Связь клиента с проектами через `client_projects`.
- Сотрудники со стороны клиента (отдельные клиентские учётки), которым выдаётся
  доступ к конкретным проектам — они видят прогресс, листы реализации и фото.
- История платежей клиента.

### 2.5 Подрядчики
- Карточки подрядных организаций.
- Привязка к проектам с условиями (сумма контракта, статус).
- Расходы по проекту автоматически могут отнестись к конкретному подрядчику.

### 2.6 Документы
- Загрузка договоров, актов, счетов и прочих файлов в облачное хранилище
  (Google Cloud Storage).
- Привязка документа к проекту.
- Прямая загрузка из браузера через Uppy + presigned URLs.
- Поддерживаются изображения, PDF и видео до 100 МБ на файл.
- Защита доступа через ACL — клиент видит только свои документы.

### 2.7 Листы реализации (Implementation Sheets)
Главный инструмент контроля выполнения работ по смете.
- Лист реализации создаётся под проект и содержит позиции из сметы.
- По каждой позиции отслеживается: количество запланированное и выполненное,
  статус (запланировано / в работе / выполнено), исполнитель, комментарии.
- К позициям прикрепляются **фото с объекта** (включая видео).
- **Полноэкранный просмотр фото** с зумом, поворотом, навигацией и горячими
  клавишами.
- Журнал изменений (`change_logs`) — кто что менял, когда и с какого значения
  на какое.
- Парсинг счёта/инвойса в позиции (загрузил PDF — система разобрала).
- Клиент может зайти и в реальном времени увидеть прогресс и фото
  (с автообновлением кэша).

### 2.8 Транспорт и фотоконтроль
- Карточки служебных автомобилей (марка, модель, гос. номер, пробег).
- **Еженедельный воскресный фотоконтроль** в 8 шагов: фото со всех ракурсов,
  одометр, состояние салона.
- На фото накладываются водяные знаки (дата, время, авто, водитель).
- Серверная проверка времени съёмки — нельзя «подложить» старое фото.
- Контроль монотонности пробега — пробег только растёт, директор может
  скорректировать с записью в журнал.
- **PDF-отчёт** по фотоконтролю автоматически складывается в облако.
- Полный аудит-лог всех изменений по транспорту.
- Доступ имеют только сотрудники с ролью «Водитель» в их HR-карточке.

### 2.9 Инструменты
- Учёт инструмента: текущий держатель, перемещения между сотрудниками
  (`tool_movements`), история выдач и возвратов.

### 2.10 Аналитика
- Расходы по категориям, по проектам, по подрядчикам.
- Финансовые сводки.
- Доступ — для директора и админа (по правам).

### 2.11 История / журнал действий
- `audit_logs` — общесистемный журнал.
- `admin_actions` — действия админов (блокировки, сбросы пароля, выдача прав).
- `permission_audit_log` — изменения прав по ролям и сотрудникам.
- `personnel_role_audit_log` — изменения роли «Водитель».
- `vehicle_audit_log` — корректировки пробега и фотоконтроль.

### 2.12 Гибкие права (Permissions Framework)
Страница `/permissions` (только админ). Четыре вкладки:

1. **«По ролям»** — список из 47 прав в 7 категориях; галочка стоит — доступ
   есть, снята — нет. Изменения применяются к роли в целом.
2. **«По сотрудникам»** — индивидуальное переопределение для конкретного
   пользователя: «По умолчанию» (наследовать от роли) / «Включено» / «Выключено».
3. **«По персоналу»** — то же самое, но через HR-карточку. Если у сотрудника
   нет учётки — кнопка-подсказка завести её прямо в карточке персонала.
4. **«Журнал»** — последние 100 изменений прав: кто, когда, какое право,
   с чего на что переключил.

Защиты:
- Право `users.manage_permissions` системное — отнять его нельзя.
- Защита от «последнего админа» — нельзя оставить систему без админов.
- Подтверждение, если пользователь снимает право у самого себя.

### 2.13 Telegram-уведомления
- Привязка чата по `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`.
- Таблица `telegram_notifications` — что было отправлено и когда.
- Используется для критичных событий (например, новые расходы, проблемы
  с фотоконтролем).

### 2.14 Email-уведомления
- Очередь `email_notifications` для отправки уведомлений на почту.

### 2.15 Админ-панель
- Управление пользователями: блокировка, разблокировка, сброс пароля,
  установка пароля, принудительный logout.
- Список активных сессий.
- Журнал попыток входа (`login_attempts`).
- Журнал действий админов.
- Системная статистика (`/api/admin/stats`).

## 3. Технические особенности

### Архитектура
- **Frontend:** React + TypeScript + Vite, маршрутизация Wouter, состояние —
  TanStack Query, формы — React Hook Form + Zod, UI — shadcn/ui на Radix,
  стили — Tailwind CSS. Иконки — lucide-react.
- **Backend:** Node.js + Express, ORM Drizzle, PostgreSQL (Neon).
  Сессии — express-session, пароли — bcrypt.
- **Хранилище файлов:** Google Cloud Storage с object-level ACL, прямые
  загрузки через presigned URLs.
- **Сборка:** Vite (фронт), esbuild (бэк).

### Мобильность и UX
- Дизайн mobile-first — приложение комфортно работает с телефона.
- Все ключевые формы (расходы, авансы, фото) оптимизированы под пальцы.
- Голосовой ввод расходов — зажал кнопку, надиктовал, система распознала.

### Двуязычность
- Русский и английский интерфейс с переключателем.

### Корпоративный дизайн «Pag CRM Corporate v1»
- Единый стиль через CSS-переменные (`--corp-bg`, `--corp-surface`,
  `--corp-ink`, `--corp-accent`, `--corp-pos/neg`, `--corp-brand`,
  `--corp-mono`, `--corp-font`).
- Общие компоненты: `CorpHeader`, `CorpEmpty`, `CorpHeroSummary`, `MoneyAED`,
  форматтеры дат и чисел.
- Постепенно мигрируется страница за страницей.

### Деплой
- Платформа Replit Deployments. Домен — pagcrm.com.
- Production-БД отдельная от dev.

## 4. Безопасность

- Все пароли — bcrypt-хеши, в открытом виде нигде не хранятся.
- Сессии в БД с возможностью принудительного завершения админом.
- Защита от брутфорса логина — журнал попыток входа.
- Каждый эндпоинт защищён `requirePermission(KEY)` — нельзя обойти UI.
- Дополнительный слой `denyWorker` блокирует пути, недоступные рабочим.
- Last-admin protection — нельзя выключить право у последнего админа
  и заблокировать систему.
- ACL на файлах — нельзя получить чужой документ напрямую через ссылку.
- Создание новых админов закрыто — только через админ-панель (ни через
  карточку персонала, ни через публичный API).
- Серверная проверка времени фото фотоконтроля — нельзя подделать дату.

## 6. Куда смотреть в коде

- `client/src/pages/` — все страницы интерфейса.
- `client/src/components/` — переиспользуемые компоненты, в т. ч. `corp-ui.tsx`.
- `server/routes.ts` — все REST-эндпоинты (~170).
- `server/storage.ts` — слой работы с БД (интерфейс `IStorage`).
- `server/lib/permissions.ts` — фреймворк гибких прав.
- `shared/schema.ts` — единая модель данных (Drizzle + Zod).
- `shared/permissions.ts` — реестр 47 прав по 7 категориям.
- `replit.md` — техническая выжимка для разработчика.
